# Phase 22a.4 Step 4: Wrapper Pooling Assessment

**Date:** 2026-03-20  
**Phase:** 22a.4 Step 4  
**Status:** Assessed and validated; no runtime pooling change made

## What was implemented

Completed the Step 4 assessment by measuring the current small-body buffered path and deciding not to pool `BufferedResponseCollectorHandler` in 22a.4.

1. Assessed the only current Step 4 candidate that exists in the codebase today.
   - `BufferedResponseCollectorHandler` is the only 22a.4 candidate presently shipping
   - `DecompressionBodySource` and `TeeBodySource` remain 22a.5 work items and were therefore not poolable candidates yet

2. Measured current small-body buffered request paths against a real loopback server.
   - ran a temporary `net8.0` loopback harness using a local `HttpListener` server and the real raw-socket client path
   - scenarios measured:
     - 1 KB JSON GET buffered roundtrip
     - 1 KB JSON POST buffered roundtrip
     - 1 KB form POST buffered roundtrip

3. Recorded the observed retained-allocation and latency results used for the pooling decision.
   - `GET 1KB JSON`: `median_bytes_per_request=14`, `median_latency_us=135.4`
   - `POST 1KB JSON`: `median_bytes_per_request=18`, `median_latency_us=153.8`
   - `POST 1KB form`: `median_bytes_per_request=4`, `median_latency_us=130.8`

4. Deferred wrapper pooling.
   - no `ObjectPool<BufferedResponseCollectorHandler>` was introduced
   - no `PrepareForPool()` reset pattern was added
   - current evidence does not show a measurable allocation problem worth the extra lifecycle complexity in 22a.4

## Decisions and trade-offs

1. **`BufferedResponseCollectorHandler` pooling is deferred.**  
   The measured retained allocations for the loopback small-body path are already effectively at the noise floor after forced GC. A single ~64-128 byte handler wrapper is not material relative to actual transport/body costs in the measured path, so pooling would add reset/reuse complexity without a demonstrated payoff.

2. **22a.5 wrapper candidates were intentionally not pre-pooled early.**  
   `DecompressionBodySource` and `TeeBodySource` do not exist yet, so Step 4 closed only the currently-real candidate instead of introducing speculative pooling infrastructure ahead of the owning features.

3. **The decision is evidence-based but local.**  
   This assessment used a local temporary loopback harness rather than the future repo-owned benchmark suite. The formal benchmark files remain planned for 22a.6, but the Step 4 decision no longer depends on guesswork.
