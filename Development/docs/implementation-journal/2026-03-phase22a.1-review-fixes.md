## What Was Fixed

This pass addressed the review findings from `Development/docs/phases/phase22a/review-22a.1-core-body-model.md` that were actionable inside this workspace:

1. Threaded the dispatch `CancellationToken` through `BufferedResponseCollectorHandler` body draining and trailer reads so buffered collection now cancels correctly on timeout/caller cancellation.
2. Closed the `AttachRequestRelease` TOCTOU race in both `UHttpStreamingResponse` and `UHttpResponse` by moving the disposed check inside the CAS registration loop.
3. Moved pooled-request response-hold transfer into the buffered and streaming bridge handoff paths, so the release callback is attached before the response task is completed.
4. Hardened `RequestBodyReadSession.Dispose()` so stream-dispose failures mark the body as faulted and block replay/reopen instead of silently clearing the active-session guard.
5. Changed `ResponseBodyStream` ownership semantics: disposing the stream no longer disposes the parent `UHttpStreamingResponse`; partial-body dispose aborts the source, while EOF dispose preserves trailers until the response itself is disposed.
6. Documented the 22a.1 handler contract explicitly: `OnResponseError` is pre-response-start only, and post-start failures must surface through `IResponseBodySource`.
7. Cached `FileRequestBody.Length` at construction time.
8. Preserved cancellation tokens inside `SingleReaderChannel<T>` pending-read cancellation and fixed terminal double-completion so the first fault wins.
9. Added finalizer fallbacks for `OwnedMemoryRequestBody` and `StreamRequestBody`, while keeping finalization armed until body resources are actually released.
10. Updated `NullHandler` to call both `Abort()` and `DisposeAsync()`.
11. Completed the 22a.1 linker audit for the new response/body abstractions and generic instantiations currently known in the phase review.
12. Removed the dead `FactoryRequestBody` catch block, added `GC.SuppressFinalize(this)` in `ResponseBodyStream.DisposeAsync()`, removed the duplicate buffered-collector completion method, documented the legacy sync-only compat shim, and clarified the single-reader test contract in `MockResponseBodySource`.
13. Marked `MockTransport.SendAsync` and `RecordReplayTransport.SendAsync` obsolete so the remaining old transport escape hatches are explicit transitional APIs instead of silent alternatives to the handler pipeline.

## Review Rubric Closure

I re-ran the repo’s required review checklists conceptually against this pass:

- **Infrastructure rubric:** closed the ownership-transfer race, request-body disposal/replay safety gap, finalizer fallback, and single-reader channel terminal-state issue.
- **Network rubric:** closed the buffered-body cancellation hole, clarified handler/body-source callback ordering, and aligned early stream disposal with explicit abort semantics instead of implicit response disposal.

## Decisions / Trade-Offs

1. **Bridge-owned request retention is now the invariant.** Both buffered and streaming response bridges attach pooled-request release callbacks before publishing the response object, eliminating outer-send-path handoff windows.
2. **`ResponseBodyStream` now behaves like a child view, not an owner.** Disposing the stream only aborts unfinished body consumption; the caller must still dispose the `UHttpStreamingResponse` to release the full response lifetime.
3. **Session-dispose failures poison the body instead of silently reopening it.** This is stricter than the happy-path replay model, but it is safer than pretending the body can still be replayed after disposal failed.
4. **The transport contract is documented rather than re-opened.** 22a.1 still assumes no `OnResponseError` after a successful `OnResponseStartAsync`; true post-start streaming faults remain a 22a.3+ transport concern.
5. **Legacy `SendAsync` on test transports remains temporarily available but explicit.** Marking the APIs obsolete keeps compatibility for existing callers without treating them as part of the preferred 22a pipeline surface.
