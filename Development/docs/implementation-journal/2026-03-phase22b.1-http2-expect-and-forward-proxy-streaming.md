# Phase 22b.1 HTTP/2 Expect-Continue and Phase 22b.2 Forward-Proxy Streaming

**Date:** 2026-03-27  
**Phase:** 22b.1 HTTP/2 `Expect: 100-continue`, 22b.2 forward-proxy streaming  
**Status:** Implemented, review fixes applied, and validated in focused local harnesses

## What was implemented

Completed two additional Phase 22b slices after the earlier core and HTTP/1.1 expect-continue work.

1. Added HTTP/2 `Expect: 100-continue` wait handling.
   - `Http2Stream` now owns an async expect-continue signal separate from the existing response-start/completion path
   - `Http2Connection.DispatchAsync(...)` now pauses between HEADERS and DATA when the request carries both a body and `Expect: 100-continue`
   - timeout handling uses `StreamingOptions.ExpectContinueTimeoutMs` and preserves the no-wait fast path for normal requests
   - the HTTP/2 read loop now treats informational `1xx` HEADERS as interim-only and does not start the final response for them
   - `RST_STREAM` / cancellation / transport failures now fault the pending expect-continue wait through the normal stream failure path

2. Made HTTP forward-proxy dispatch lease-based and streaming-safe.
   - non-CONNECT proxy requests now use the same `DispatchOnLeaseAsync(...)` path as direct HTTP/1.1 requests
   - this keeps the proxy connection lease alive until the response body source decides whether to drain, reuse, or close it
   - streaming responses through plain HTTP proxies can now be disposed early without the proxy lease being disposed at method scope

3. Added focused validation for both slices.
   - HTTP/2 expect-continue now has coverage for interim `100`, early final response, timeout fallback, and `RST_STREAM` during the wait
   - HTTP/1.1 expect-continue now also covers keep-alive rejection reuse, late-`100` timeout sequencing, and body-send failure surfacing
   - forward-proxy coverage now includes streamed request-body absolute-form serialization and streaming-response reuse on the same proxy connection

4. Applied the Phase 22b.1 review follow-up fixes.
   - extracted shared expect-continue header/body detection into `Runtime/Transport/Internal/ExpectContinueHelper.cs`
   - HTTP/2 timeout fallback now clears the pending wait source before DATA send continues
   - HTTP/1.1 expect-continue no longer treats a body-send failure as success when a final response head also arrives; the original send exception is re-surfaced and response-read diagnostics are attached for observability
   - serializer XML docs now state that header-only/body-only methods do not flush and that callers own the flush points

## Decisions and trade-offs

1. Kept the HTTP/2 expect-continue signal on `Http2Stream` instead of folding it into response start/completion state.
   - this keeps interim `100` handling out of the final response state machine and lets read-loop failures propagate through one stream-owned signal point

2. Ignored interim HTTP/2 `1xx` HEADERS instead of materializing them into the public response surface.
   - this matches the current Phase 22b requirement and preserves the existing single-final-response handler contract

3. Restricted the proxy streaming fix to forward proxies for this slice.
   - plain HTTP proxy reuse is safe with the existing pool key because one proxy connection can serve multiple origins via absolute-form requests
   - CONNECT tunnels still require separate work for lease rebinding / tunnel identity before they can be returned safely to the pool

4. Chose transport-failure precedence over silent final-response success when the request body fails mid-send.
   - RFC 9110 permits a server to send a final response without consuming the full body, but for TurboHTTP this created a worse outcome for non-replayable uploads: partial data could look like success
   - the transport now surfaces the original body-send exception and records response-read diagnostics when both failure paths race

5. Documented the current HTTPS proxy limitation explicitly.
   - forward HTTP proxy requests honor expect-continue waits
   - CONNECT tunnel requests still route through the stream-only path, so tunneled HTTPS requests currently send the `Expect` header without a transport-level wait
   - this remains deferred to Phase 22b.2 tunnel lease/reuse work
