# Phase 22a.2 Review Fixes

Date: 2026-03-19

## What was implemented

This pass closed the review findings from `Development/docs/phases/phase22a/review-22a.2-http11-streaming.md` across the HTTP/1.1 transport, parser, disposal path, and validation surface:

1. Fixed the streaming-response lease ownership failure path in `RawSocketTransport` so handler callback failures abort the newly created `Http11ResponseBodySource` before the exception escapes. `DispatchOnLeaseAsync(...)` no longer returns a vestigial `bool`; successful completion itself is now the ownership handoff.
2. Reworked `Http11ResponseBodySource` hot-path cancellation handling so linked cancellation tokens are cached per caller token instead of allocating a new linked CTS on every body read. The same type now enforces `MaxResponseBodySize` for read-to-end bodies, synchronizes `_remainingContentLength` reads, defers timeout-message formatting to throw time, and keeps the single-consumer expectation explicit.
3. Removed the chunk-size parsing dependency on `NumberStyles.HexNumber` from the streaming path by switching both buffered and streaming chunk readers to manual hex parsing. This closes the IL2CPP stripping concern by removing the risky path instead of preserving it.
4. Tightened response framing correctness in `Http11ResponseParser`:
   - `Transfer-Encoding` now requires a real terminal `chunked` token, not any suffix ending in `chunked`
   - dual `Transfer-Encoding` + `Content-Length` responses record a transport diagnostic event in `RequestContext` instead of using `Debug.WriteLine`
5. Simplified keep-alive lease reuse by making `ConnectionLease.ReturnToPool()` release the semaphore permit itself. `Http11ResponseBodySource.CompleteBody()` now either returns the connection to the pool or disposes it, instead of depending on a fragile `ReturnToPool()` plus `Dispose()` pairing.
6. Replaced the tiny pooled chunk-header buffer in `Http11RequestSerializer` with a local `byte[18]` buffer, avoiding the low-value `ArrayPool` rent/return pair on that path.
7. Fixed Unity/package compatibility blockers encountered while validating the review pass:
   - `LegacyHttpHandlerCompatExtensions` now uses the available `ReadOnlySequence<byte>.ToArray()` shape that Unity 2021.3 accepts
   - `TurboHTTP.Auth` regained access to Core’s shared-content request clone helper via `InternalsVisibleTo`
   - the editor monitor replay path now uses `UHttpClient.SendBufferedAsync(...)`
   - several runtime test files now use the repo’s `AssertAsync` helper instead of `Assert.ThrowsAsync`, which Unity’s bundled NUnit surface does not provide
8. Removed the remaining steady-state allocations found in the HTTP transport hot paths after the review fix pass:
   - chunked request serialization now rents its 18-byte chunk-header buffer from `ArrayPool<byte>.Shared` instead of allocating a new array per request
   - fixed-length `Content-Length` header emission now writes the `long` directly into the pooled header writer instead of allocating via `ToString(...)`
   - chunked response parsing no longer allocates a temporary 2-byte array for every chunk terminator
   - `SaeaStream` and `PollSelectStream` now use pooled fallback buffers when a caller supplies non-array-backed `Memory<byte>` to `ReadAsync(...)`

## Decisions and trade-offs

1. Linked read-token caching is keyed by the caller token, not created once in the constructor. `IResponseBodySource.ReadAsync(...)` accepts a token per call, so a single constructor-time linked CTS would be incorrect if a consumer changes tokens across reads.
2. The dual-framing warning moved to `RequestContext.RecordEvent(...)` because the transport already exposes request-scoped diagnostics there and it survives release builds, unlike `Debug.WriteLine`.
3. The chunked early-dispose budget remains decoded-byte-based. That was already the specified behavior in 22a.2; the doc now says so explicitly instead of leaving the wire-byte distinction implicit.
4. The record/replay streaming-body limitation is documented, not fully solved here. `RecordReplayTransport` still records bodies only from `TryGetBufferedData(...)`, so streaming HTTP/1.1 responses replay as empty bodies until the record/replay path grows streaming capture support.
5. The non-array-backed `Memory<byte>` fallback on `SaeaStream`/`PollSelectStream` is still a rare path in the current codebase, but it is a transport boundary that can be hit by BCL consumers such as TLS wrappers. Pooling there removes the last guaranteed GC allocation from that fallback path without changing the public contract.
