# Phase 22b.3: HTTP/1.1 Response Trailer Parsing

**Date:** 2026-03-21  
**Phase:** 22b.3 response trailers  
**Status:** Implemented and validated in a focused local compile/run harness

## What was implemented

Completed the first Phase 22b slice by landing the full HTTP/1.1 response-trailer path described by 22b.3:

1. Added shared prohibited-trailer filtering in Transport.
   - introduced `Runtime/Transport/Internal/TrailerFieldValidator.cs`
   - populated comprehensive response/request deny lists aligned with RFC 9110 Section 6.6.2 categories
   - kept the shared request set in place for planned 22b.4 reuse

2. Replaced HTTP/1.1 chunked-trailer discard behavior with parsing/exposure.
   - `Http11ResponseParser` now parses trailer lines instead of discarding them
   - malformed trailer lines without a usable field name are skipped defensively
   - prohibited trailers are silently filtered
   - embedded CR/LF in trailer values now fail parsing
   - the buffered parser path now preserves parsed trailers on `ParsedResponse`

3. Extended `Http11ResponseBodySource` to publish trailers for streaming responses.
   - chunked responses now allocate a deferred `TaskCompletionSource<HttpHeaders>` only when needed
   - `GetTrailersAsync(...)` now returns parsed chunked trailers after EOF
   - non-chunked responses still avoid a trailer TCS and return `HttpHeaders.Empty`
   - abort/dispose-before-EOF now faults pending trailer retrieval with `ObjectDisposedException`
   - body read faults now propagate through trailer retrieval instead of being hidden

4. Propagated trailers through buffered response collection.
   - `BufferedResponseCollectorHandler` now captures trailers after body drain and passes them into `UHttpResponse`
   - `UHttpResponse` now exposes an additive `Trailers` property
   - the public constructor surface stays backward-compatible via a defaulted optional `trailers` parameter
   - the legacy buffered transport path in `RawSocketTransport` now preserves parsed trailers instead of dropping them

5. Completed the wrapper audit gate for `IResponseBodySource`.
   - verified `ObservedResponseBodySource`, `DecompressionBodySource`, `TeeBodySource`, `PluginContext.ObservedBodySource`, `BufferedResponseBodySource`, `MockResponseBodySource`, and `Http2ResponseBodySource`
   - no wrapper code changes were needed in this slice because the relevant delegation logic was already present

## Decisions and trade-offs

1. Preserved the existing "drain on trailer retrieval" behavior for active non-chunked HTTP/1.1 body sources.
   - This keeps `GetTrailersAsync(...)` compatible with existing callers that used it as a drain barrier, while still avoiding the new trailer TCS allocation on non-chunked responses.

2. Kept trailer-line parsing logic lightweight and local to the HTTP/1.1 parser/body-source path.
   - `TrailerFieldValidator` owns the RFC deny lists.
   - field-name/value validation remains transport-local instead of broadening Core surface area in this slice.

3. Avoided new module dependencies and asmdef changes.
   - the work stayed within existing `TurboHTTP.Core` and `TurboHTTP.Transport` boundaries plus runtime tests
   - no optional-module coupling was introduced

4. Did not add new `link.xml` preserves yet for `TaskCompletionSource<HttpHeaders>`.
   - this generic is referenced directly from managed code, so no linker workaround was added blindly
   - IL2CPP verification remains a required follow-up validation item from the 22b docs
