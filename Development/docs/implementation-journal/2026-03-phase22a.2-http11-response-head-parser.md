# Phase 22a.2 Step 4: HTTP/1.1 Response Head Parser Split

**Date:** 2026-03-18  
**Phase:** 22a.2 Step 4  
**Status:** Implemented and validated in a runtime harness; Unity/package validation still pending

## What Was Implemented

Completed the parser split that prepares HTTP/1.1 for streamed response bodies:

1. `Http11ResponseParser` now has a header-only entry point:
   - `ParseHeadAsync(Stream, HttpMethod, CancellationToken)`
   - consumes interim `1xx` responses before exposing the final head
   - determines framing without buffering the body
2. Added `ParsedResponseHead` as the staged parse result, carrying:
   - final status code
   - response headers
   - keep-alive decision
   - resolved body framing kind
   - validated content length when applicable
   - ownership of the parser’s `BufferedStreamReader`
3. Added `Http11ResponseBodyKind` with the four HTTP/1.1 framing modes needed by later body sources:
   - `Empty`
   - `Chunked`
   - `ContentLength`
   - `ReadToEnd`
4. Preserved the existing buffered parser API by rebuilding `ParseAsync(...)` on top of:
   - `ParseHeadAsync(...)`
   - `CompleteBufferedResponseAsync(...)`
5. `BufferedStreamReader` is now transport-internal instead of parser-private and exposes a `Memory<byte>`-based read path so the next step can hand it directly to `Http11ResponseBodySource`.
6. Added parser regressions covering:
   - HEAD responses resolve to empty-body framing even with `Content-Length`
   - `Transfer-Encoding` takes precedence over `Content-Length`
   - staged parsing preserves prefetched body bytes when the buffered completion path runs afterward

## Decisions / Trade-Offs

1. **Buffered compatibility was kept intentionally.** Existing `ParseAsync(...)` callers still work, which lets the transport move to streaming incrementally instead of forcing a single large parser/transport rewrite.
2. **`Transfer-Encoding` now wins framing without sanitizing response headers.** When both `Transfer-Encoding` and `Content-Length` are present, the parser logs a warning and uses `Transfer-Encoding` framing, but it leaves the original headers intact for now.
3. **`Content-Length: 0` remains a fixed-length body kind for normal responses.** `Empty` is reserved for no-body semantics driven by method/status (`HEAD`, `1xx`, `204`, `304`, `101`) so Step 5 can map protocol-mandated empty responses cleanly.
4. **The reader transfer is explicit even before the body source exists.** `ParsedResponseHead` already owns the `BufferedStreamReader`, which avoids another parser refactor when Step 5 introduces `Http11ResponseBodySource`.
