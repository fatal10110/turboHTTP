# Phase 22a.4 Review Remediation

**Date:** 2026-03-20  
**Phase:** 22a.4 review remediation  
**Status:** Implemented and validated in the local compile/run harness

## What was implemented

Remediated the 22a.4 review findings across the buffered fast-path runtime and test surface.

1. Made detached buffered ownership transfer copy-safe.
   - `DetachedBufferedBody` is now a `readonly struct`.
   - owner transfer/disposal state moved behind a shared reference-type token so `DetachOwner()` and `DisposeOwnedResources()` remain safe across defensive copies.
   - added direct tests covering readonly-field copy scenarios.

2. Blocked detach for buffered sources that still own trailers.
   - `BufferedResponseBodySource.TryDetachBufferedBody(...)` now returns `false` when trailers are present.
   - `MockResponseBodySource` now matches that behavior for test-only buffered sources.

3. Hardened the HTTP/1.1 empty-body detach case.
   - `Http11ResponseBodySource` now tracks `_hasReadData` and rejects detach after any `ReadAsync(...)` or `DrainAsync(...)` call.
   - the detach predicate no longer depends on `Interlocked.Read(ref long)` for `_remainingContentLength`.
   - documented that the empty-body detach case is only valid because `CompleteBody()` already returned the lease in the constructor path.

4. Removed duplicate streaming-default definitions and tightened supporting behavior.
   - `StreamingOptions` now uses shared default constants for field initialization and `IsDefault()`.
   - property validation now reports the actual property name instead of the generic `"value"` parameter name.
   - `Http11RequestSerializer` now normalizes `streamingOptions ?? new StreamingOptions()` at method entry instead of carrying duplicate fallback constants.
   - added doc text clarifying that the effective HTTP/2 per-stream receive buffer may be raised to satisfy connection requirements.

5. Tightened cleanup/documentation edges called out by review.
   - `MockResponseBodySource.DisposeAsync()` is now idempotent after detach.
   - `ResponseCollectorHandler` now documents why `detachedOwner.Dispose()` plus `detachedBody.DisposeOwnedResources()` is safe.
   - `Http2Connection` backward-compatible constructor comments now explicitly document the default options/streaming-options behavior.
   - `Http2ResponseBodySource` now documents that detach still flows through the normal connection-level consumption/WINDOW_UPDATE trigger.

6. Added missing review coverage.
   - buffered detached sequence collector test now asserts body content equals `"payload"`.
   - added direct tests for:
     - readonly-copy `DetachedBufferedBody` owner transfer/disposal
     - buffered-source trailer detach rejection
     - mock dispose idempotency after detach
     - HTTP/1.1 detach rejection after read/drain attempts on empty-body sources

## Decisions and trade-offs

1. **Did not make `DetachedBufferedBody` internal.**  
   The review's suggested visibility change conflicts with the public `IResponseBodySource.TryDetachBufferedBody(...)` contract. Instead, the fix removed the actual bug source by making the struct immutable and moving owner-transfer state into a shared reference token. This preserves the public interface shape while eliminating defensive-copy leaks.

2. **Trailer-bearing buffered sources now prefer the fallback collector path.**  
   This keeps detach limited to ownership shapes that are already safe under the current buffered collector behavior without expanding `DetachedBufferedBody` to carry trailer state.

3. **HTTP/2 connection-window behavior needed clarification, not new transport code.**  
   The detach path already routes released flow-controlled bytes through `OnResponseBytesConsumed(...)`, which fires the existing connection-level `MaybeSendConnectionWindowUpdateAsync(...)` path. I documented that behavior instead of adding a duplicate trigger.
