# Phase 22a.2 Steps 7-8: HTTP/1.1 Streaming Response Transport Handoff and Timeout Scope

**Date:** 2026-03-18  
**Phase:** 22a.2 Steps 7-8  
**Status:** Implemented and validated in a runtime harness; Unity/package validation still pending

## What Was Implemented

Completed the direct transport wiring that makes the HTTP/1.1 response stream live beyond header receipt:

1. `RawSocketTransport` now routes direct HTTP/1.1 responses through the staged parser path:
   - serialize request
   - parse response head with `Http11ResponseParser.ParseHeadAsync(...)`
   - construct `Http11ResponseBodySource`
   - deliver status/headers/body source through `IHttpHandler.OnResponseStartAsync(...)`
2. The direct HTTP/1.1 lease is now retained by the response body source until one of three terminal outcomes:
   - full body consumption
   - successful async dispose/drain within policy
   - abort/close on cancellation, error, or over-budget disposal
3. Buffered and streaming public APIs now share the same direct HTTP/1.1 response path:
   - buffered callers consume the live body source through `BufferedResponseCollectorHandler`
   - streaming callers receive the live body source through `StreamingDispatchBridge`
4. Added `TransportBehaviorFlags.StreamingResponseRequested`, set by:
   - `BufferedDispatchBridge` to `false`
   - `StreamingDispatchBridge` to `true`
5. Request timeout scoping now differs by caller mode on the direct HTTP/1.1 path:
   - buffered callers keep the transport timeout active during body collection
   - streaming callers stop applying the request timeout once headers are delivered, so body reads are governed by the consumerâ€™s own cancellation token
6. Added focused regressions covering:
   - streaming body reads continue past the original request timeout once headers have arrived
   - buffered collection still times out when body bytes stall after headers
   - direct HTTP/1.1 reuse/close behavior remains correct after lease handoff to the live body source

## Decisions / Trade-Offs

1. **Direct HTTP/1.1 now has one response implementation for both buffered and streaming consumers.** This keeps framing, keep-alive, and failure semantics in one place instead of maintaining separate buffered and streamed receive paths.
2. **Timeout scoping changed only for direct streaming HTTP/1.1 bodies.** Buffered callers still use request timeout enforcement during body collection, and proxy HTTP/1.1 remains on the older buffered path per the phase non-goal.
3. **HTTP/1.1 lease ownership transfers by suppressing outer disposal, not by calling `ConnectionLease.TransferOwnership()`.** The body source still needs to `ReturnToPool()` on successful completion, which is incompatible with the HTTP/2-style transfer mechanism that permanently detaches the connection from the lease.
4. **Proxy streaming remains deferred intentionally.** `DispatchViaProxyAsync(...)` still uses the buffered stream path; this phase only completes the direct HTTP/1.1 transport path.
