# Phase 19a.3 — Segmented Sequences

**Date:** 2026-02-23
**Status:** COMPLETE (all specialist reviews passed)

## What Was Implemented

Eliminated copy amplification in HTTP/1.1 chunked and read-to-end body parsing by accumulating response body data into a `SegmentedBuffer` (linked chain of 16 KB `ArrayPool<byte>`-backed segments) instead of growing a single contiguous `byte[]`. The flattened `Body` property now performs a lazy one-time copy on first access; callers that only need streaming access can use `BodySequence` without ever flattening.

Four implementation steps:

1. **`SegmentedBuffer` and `SegmentedReadStream` infrastructure** (carried forward from Phase 19a.2 — already complete and reviewed).
2. **HTTP/1.1 parser rewrite** — `ReadChunkedBodyAsync` and `ReadToEndAsync` now return `SegmentedBuffer` instead of `byte[]`; `ParseAsync` routes to the correct `UHttpResponse` constructor.
3. **`UHttpResponse` segmented body support** — lazy-flatten `Body` getter, `internal BodySequence` accessor, second internal constructor accepting `ReadOnlySequence<byte>` + `IDisposable` owner.
4. **WebSocket `MessageAssembler` (deferred)** — `TurboHTTP.WebSocket` references only `TurboHTTP.Core`; adding a Performance reference would violate module isolation. Current single-pooled-rent + `BlockCopy` is already zero steady-state allocation. Deferred to a future pass.

## Decisions Made

### Assembly boundary — passing segmented body through Core
`TurboHTTP.Core` has zero external assembly references by design. `SegmentedBuffer` lives in `TurboHTTP.Performance`. To cross this boundary without adding a dependency, the internal constructor accepts BCL types only: `ReadOnlySequence<byte>` (from `System.Buffers`, already available in .NET Standard 2.1) and `IDisposable`. The transport (which references both Core and Performance) passes `SegmentedBuffer` as the `IDisposable` owner, satisfying both the assembly constraint and the disposal contract.

### `BodySequence` visibility — `internal` not `public`
`ReadOnlySequence<byte>` is a struct (copyable value type). If made `public`, callers could trivially store a copy that outlives `Dispose()` and read dangling pooled memory with no exception. Internal visibility limits consumers to within-assembly code that can be audited for correct lifetime discipline.

### Lazy-flatten semantics
- Callers that access `Body` pay one `ToArray()` allocation but get a safe, lifetime-independent `byte[]`.
- Callers inside TurboHTTP that need streaming access (e.g., decompression middleware) can consume `BodySequence` zero-copy within the synchronous scope.
- The flatten is guarded by an atomic exchange so that concurrent `Dispose()` and `Body` accesses do not double-return pooled segments.

### OOM safety in `Body` getter
If `seqSnapshot.ToArray()` throws (OOM), `_segmentedBodyOwner` has already been nulled by the exchange. Without a restore, `Dispose()` would see null and skip the owner, causing a permanent pool leak (no finalizer on `SegmentedBuffer`). Fix: wrap `ToArray()` in `try/catch` and restore via `Volatile.Write(ref _segmentedBodyOwner, owner)`. The `Volatile.Write` ensures the restore is visible to a subsequent `Dispose()` call even on ARM IL2CPP.

### ARM IL2CPP memory-ordering
`Interlocked.Exchange` provides a full memory fence on ARM64 IL2CPP, correctly ordering the owner-claim in both the `Body` getter and `Dispose`. `Volatile.Write` in the OOM path ensures the restored owner pointer is visible across threads without the full fence overhead of a second `Interlocked` call.

### WebSocket deferral
`TurboHTTP.WebSocket` references only `TurboHTTP.Core`. `MessageAssembler` uses a single `ArrayPool<byte>` rent + `BlockCopy` loop, which is already O(1) allocations per reassembled message. Adding a Performance reference for marginal gain would violate the module-isolation rule. Deferred.
