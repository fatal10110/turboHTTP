# Phase 19a.2 — Buffer-Writer-First Serialization

**Date:** 2026-02-23
**Session:** Context-resumed from previous session

## What Was Implemented

Phase 19a.2 introduces a zero-copy serialization path for request bodies. Instead of `byte[]` intermediates, JSON and multipart builders write directly into pooled `ArrayPool<byte>` buffers and transfer ownership to the transport via `IMemoryOwner<byte>`. The transport disposes the owner exactly once after the body is on the wire.

## Files Created

### `Runtime/Performance/PooledArrayBufferWriter.cs`
Public `IBufferWriter<byte>` implementation backed by `ArrayPool<byte>.Shared`. Key features:
- `GetSpan(sizeHint)` / `GetMemory(sizeHint)` / `Advance(count)` — full `IBufferWriter<byte>` contract
- `DetachAsOwner()` — transfers buffer ownership to `ArrayPoolMemoryOwner<byte>`, sets writer to detached state
- `Reset()` — resets write position without returning buffer (for writer reuse)
- Growth algorithm: double capacity up to 1MB, then +50%; `MaxBufferSize = 64MB`
- `ThrowIfDetachedOrDisposed()` with `[MethodImpl(AggressiveInlining)]`

### `Runtime/Core/Internal/PooledArrayBufferWriter.cs`
Internal copy for `TurboHTTP.Core.Internal` namespace. Required because `TurboHTTP.JSON` and `TurboHTTP.Files` reference only `TurboHTTP.Core` — they cannot reference `TurboHTTP.Performance`. Sync comment references canonical Performance version.

### `Runtime/Core/Internal/ArrayPoolMemoryOwner.cs`
Internal copy of `ArrayPoolMemoryOwner<T>` for use by `Core.Internal.PooledArrayBufferWriter`. Sync comment references canonical `Runtime/Performance/ArrayPoolMemoryOwner.cs`.

### `Runtime/Transport/RawSocketTransport.cs`
- `SendOnLeaseAsync` / `SendOnStreamAsync`: `try/finally { request.BodyOwner?.Dispose(); }` wraps body serialization
- `SendViaProxyAsync` HTTP path: `try/finally { request.BodyOwner?.Dispose(); }` after `PrepareHttpProxyForwardRequest`
- `SendViaProxyAsync` HTTPS path: `try/finally { request.BodyOwner?.Dispose(); }` starts BEFORE `EstablishConnectTunnelAsync` so tunnel failures (407, network error) don't leak the owner

### `Runtime/Transport/Http2/Http2Connection.cs` — `SendRequestAsync`
Restructured to two-level try/finally:
1. **Outer `try/finally`**: wraps entire method including the `lock` block — disposes `request.BodyOwner` on all exit paths including GOAWAY/MAX_CONCURRENT_STREAMS exceptions thrown inside the lock, and post-lock cancellation
2. **Inner `try/finally`**: stream cleanup with `if (stream != null)` guard
3. **Body-send `try/finally`**: disposes `BodyOwner` promptly after body is on the wire (or on send failure)

Double-dispose is safe: `ArrayPoolMemoryOwner.Dispose()` is idempotent (null-swap pattern).

## Decisions Made

### `Body = bodyOwner.Memory.ToArray()` — defensive copy
`UHttpRequest.Body` is a `byte[]` heap copy of the pooled buffer, not a view into it. This enables:
- The stale-connection retry path to re-send from `request.Body` after `BodyOwner` is disposed
- Middleware `With*` transforms to create new requests without re-calling `ToArray()`
- HTTP/2 and HTTP/1.1 transports to read from `Body` independently of owner lifetime

### Internal copies instead of shared assembly
`TurboHTTP.JSON` and `TurboHTTP.Files` cannot reference `TurboHTTP.Performance` (only Core is allowed). Solution: `Runtime/Core/Internal/` contains internal copies of `PooledArrayBufferWriter` and `ArrayPoolMemoryOwner` with sync comments. Same pattern used by `TurboHTTP.WebSocket`.

### Outer try/finally over lock block
The pre-flight exception paths (GOAWAY, MAX_CONCURRENT_STREAMS, post-lock cancellation) exit before the inner `try` that normally disposes `BodyOwner`. The outer `try/finally` is the correct fix — explicit `bodyOwnerDisposed` flag was not needed due to idempotent dispose.

### HTTPS proxy try/finally placement
The `try/finally { request.BodyOwner?.Dispose(); }` in `SendViaProxyAsync` must begin before `EstablishConnectTunnelAsync`, not after it. A failed CONNECT handshake (407, IO error) exits before `PrepareHttpsProxyTunnelRequest` is called, and would leak the owner if the `try` started after tunnel establishment.

## Known Acceptable Patterns

- **Double-dispose in HTTP/2**: Body-send `finally` + outer `finally` both call `request.BodyOwner?.Dispose()`. Safe and intentional — `ArrayPoolMemoryOwner.Dispose()` is idempotent.
- **Stale-connection retry**: First `SendOnLeaseAsync` disposes `BodyOwner`; retry re-uses `request.Body` (the `byte[]` copy). Second `SendOnLeaseAsync` calls `BodyOwner?.Dispose()` again (null after first dispose — no-op).
- **LiteJson bridge allocations**: Two unavoidable allocations (string + byte[]) in the `IBufferWriter<byte>` bridge. Documented in XML. Eliminable in a future version when LiteJson gains a span-based writer.
