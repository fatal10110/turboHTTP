# Phase 22a.4 Step 5: Allocation and Latency Audit

**Date:** 2026-03-20  
**Phase:** 22a.4 Step 5  
**Status:** Audited and validated; no additional tuning change required in 22a.4

## What was implemented

Completed the 22a.4 Step 5 audit by walking the hot paths, checking the new buffered fast-path behavior against the intended memory model, and confirming there was no concrete tuning change justified by the local evidence gathered in this phase.

1. Audited the small buffered request hot path.
   - HTTP/1.1 small buffered bodies now stay off `RequestBodyReadSession`
   - small buffered HTTP/1.1 bodies within the configured threshold coalesce into a single `headers + body` write
   - larger buffered HTTP/1.1 bodies still bypass the streaming session path and write body bytes directly from `ReadOnlyMemory<byte>`
   - HTTP/2 buffered request sending still writes directly from buffered memory into DATA frames with no read-session wrapper

2. Audited the buffered response hot path.
   - detachable buffered sources now transfer ownership directly into `UHttpResponse`
   - non-detachable buffered sources still take the existing copy/drain path into `SegmentedBuffer`
   - normal socket-backed HTTP/1.1 responses remain on the drain path because `Http11ResponseBodySource` owns a live reader, not a finished in-memory payload

3. Audited the streaming paths for new per-chunk allocation risks.
   - HTTP/1.1 request streaming still uses one read session and one rented transfer buffer per request
   - HTTP/2 request streaming still uses one rented send buffer per request
   - HTTP/2 response streaming still keeps body chunks in pooled byte arrays inside the bounded queue; 22a.4 did not add new per-chunk object allocation requirements on the active streaming read path

4. Closed the tuning pass with no additional runtime edits.
   - no inline/tuning patch was required after the local audit
   - no extra pooling or `ValueTask`-shape rewrite was added in this step
   - the local loopback measurements from Step 4 did not show a small-body buffered-path regression signal strong enough to justify speculative tuning in 22a.4

## Decisions and trade-offs

1. **No extra tuning patch was added without a measured problem.**  
   The Step 4 loopback measurements showed the small buffered GET/POST path retaining only single-digit to low-double-digit bytes per request after forced GC. That is not a credible signal for adding more lifecycle or fast-path complexity in 22a.4 just to chase theoretical wins.

2. **HTTP/1.1 loopback remains copy/drain-based by design in 22a.4.**  
   This is not a regression from the intent of the current implementation; it follows directly from the fact that `Http11ResponseBodySource` still owns a live socket reader. The detach fast path added in Step 3 is therefore primarily a buffered-source and HTTP/2 win in this phase.

3. **Formal regression benchmarking still belongs to 22a.6.**  
   Step 5 closes with a local audit and local loopback evidence, but the repo-owned benchmark suite and cross-platform benchmark artifact generation are still deferred to the final validation phase.
