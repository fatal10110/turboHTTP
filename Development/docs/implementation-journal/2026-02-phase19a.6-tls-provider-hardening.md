# Phase 19a.6 — TLS Provider Hardening

**Date:** 2026-02-27
**Status:** COMPLETE (2 review rounds, all findings resolved)

---

## What Was Implemented

Phase 19a.6 hardens the TLS provider selection layer with clearer behavioral contracts, a diagnostic observability hook, reduced per-connection allocation, and a platform capability diagnostics API.

### Step 0: `TlsBackend.Auto` Documentation Clarification

Updated the `TlsBackend.Auto` XML doc comment to explicitly state the fallback boundary:

- **Fallback to BouncyCastle occurs only** for platform capability exceptions: `PlatformNotSupportedException`, `TypeLoadException`, `MissingMethodException`, etc.
- **Certificate validation failures, authentication errors, hostname mismatches, and network errors do NOT trigger fallback** — they propagate as exceptions to the caller.

This prevents user confusion about whether a self-signed cert or expired cert rejection would silently switch providers.

### Step 1: `TlsProviderSelector.DiagnosticLogger`

Added `DiagnosticLogger` static property (`Action<string>`) to `TlsProviderSelector`. Receives one-line messages at key TLS provider transition events:

- `MarkSslStreamViable()` — logs once on the first successful SslStream handshake (0→1 transition via CAS)
- `MarkSslStreamBroken()` — logs once on the first platform capability failure (→2 transition; only if prev ≠ 2)
- `GetAutoProvider()` — logs when SslStream is unavailable at initialization-time and BC fallback is used

Implementation detail: backed by a `volatile` field (`_diagnosticLogger`) with `Volatile.Read`/`Volatile.Write` accessors. This provides ARM IL2CPP memory barrier guarantees without a per-read lock (a plain auto-property would allow stale null reads on ARM). Pattern matches `MonitorMiddleware.DiagnosticLogger` in the Observability assembly.

Logging fires after the Interlocked state operation. Under the specific concurrent scenario where two threads both read `prev ≠ 2` before either writes, duplicate logging of the "broken" message is theoretically possible. This is cosmetic-only and explicitly documented in the comment.

### Step 2: Static Delegate Caching in `SslStreamTlsProvider`

Changed `ValidateServerCertificate` from an instance method to a `static` method. Added `s_validateCertCallback` static readonly field caching the delegate:

```csharp
private static readonly RemoteCertificateValidationCallback s_validateCertCallback =
    new RemoteCertificateValidationCallback(ValidateServerCertificate);
```

`SslStream` construction now passes `s_validateCertCallback` instead of `new RemoteCertificateValidationCallback(ValidateServerCertificate)` per call. Eliminates one delegate allocation per TLS handshake.

Added explicit "no silent downgrade" comment: the validator accepts only `SslPolicyErrors.None` — certificate failures, hostname mismatches, and untrusted roots all propagate as `AuthenticationException`.

Also narrowed the `Expression.Lambda.Compile()` catch in the static constructor from bare `catch` to `catch (PlatformNotSupportedException)` + `catch (NotSupportedException)`. This ensures AOT/JIT restriction failures are silently absorbed while unexpected exceptions (null references, etc.) surface properly.

Added documentation comment explaining `checkCertificateRevocation: false` in the non-ALPN auth path is intentional: OCSP/CRL checks are unreliable on mobile networks and the BCL default for `SslClientAuthenticationOptions` is also false.

### Step 3: `TlsPlatformCapabilities` (pre-existing — confirmed complete)

`TlsPlatformCapabilitySummary` struct and `TlsPlatformCapabilities` static class already existed with full implementation:
- `IsSystemTlsAvailable`, `SystemProviderDescription`, `IsSystemAlpnExpected`, `IsBouncyCastleAvailable`, `IsSystemTlsKnownBroken`, `RecommendedBackend`
- `GetSummary()`, `GetDiagnosticSummary()`, `RefreshForTesting()`
- `Lazy<T>(ExecutionAndPublication)` caching

## Decisions Made

### `DiagnosticLogger` backed by `volatile` field
Matches the pattern used by `MonitorMiddleware.DiagnosticLogger` in the same codebase. A plain auto-property generates a compiler-backed field with no memory ordering guarantee. On ARM IL2CPP (iOS/Android), a main-thread write to the logger may remain invisible to a connection pool thread unless a barrier exists. `Volatile.Read/Write` provides the release-acquire fence. Pattern choice: `volatile` field + explicit property (not `Volatile.Read` inline in `Log()`) for clarity and consistency with codebase conventions.

### `s_cached` volatile in `TlsPlatformCapabilities`
`RefreshForTesting()` swaps the `Lazy<T>` reference under a lock, but `GetSummary()` reads without a lock. On ARM IL2CPP, a thread calling `GetSummary()` concurrently with `RefreshForTesting()` may observe a stale `s_cached` reference. `volatile` on the field provides pointer-atomic read visibility without a lock on every `GetSummary()` call. `Lazy<ExecutionAndPublication>` is thread-safe for concurrent `.Value` reads once a reference is observed.

### `SelectRecommendedBackend` returns `TlsBackend.Auto` (not `SslStream`) for no-viable case
When `systemTlsKnownBroken == true` AND `bouncyCastleAvailable == false`, returning `TlsBackend.SslStream` produced a contradictory diagnostic: "Recommended=SslStream" with "KnownBroken=true". Returning `TlsBackend.Auto` as a "no confirmed viable backend" sentinel means any code that acts on the recommendation will probe and fail with a clear `PlatformNotSupportedException` rather than silently returning a broken provider. `TlsPlatformCapabilities` is diagnostics-only (never used in the hot path), so this change affects developer visibility only.

### `MarkSslStreamViable` guarded with provider name check in `TcpConnectionPool`
The original call was guarded only by `_tlsBackend == TlsBackend.Auto`. When Auto resolves to BouncyCastle (because `IsSslStreamKnownBroken()` returned true), the call would attempt `CompareExchange(ref state, 1, 0)` after a BouncyCastle handshake — not incorrect because CAS fails when state = 2, but semantically wrong (BouncyCastle success should not record "SslStream is viable"). Added `&& tlsResult.ProviderName == "SslStream"` to make intent explicit and robust against future state machine changes.

### Removed alias test methods
`ForceBouncyCastle_ReturnsCorrectProvider` and `ForceBouncyCastle_WhenNotAvailable_ThrowsException` called existing test methods directly, bypassing `[SetUp]`/`[TearDown]`. This left dirty state if the delegated test failed mid-way. Removed both aliases; the original tests provide full coverage.

### Expression.Compile() catch narrowed to platform exceptions
The bare `catch {}` in the static constructor swallowed all exceptions from `Expression.Lambda<Func<object>>().Compile()`. Changed to `catch (PlatformNotSupportedException)` + `catch (NotSupportedException)` to match the documented AOT/JIT restriction scenarios. Unexpected exceptions (null reference, etc.) now surface rather than silently triggering the ConstructorInfo.Invoke fallback.
