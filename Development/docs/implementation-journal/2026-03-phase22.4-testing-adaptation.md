# Phase 22.4: Testing Adaptation

**Date:** 2026-03-10  
**Phase:** 22.4 (testing adaptation)  
**Status:** Test infrastructure and legacy test-surface migration implemented; Unity compile/runtime execution still pending

## What Was Implemented

Phase 22.4 finished the remaining test-side migration work for the interceptor/handler architecture introduced in 22.1-22.3.

The main changes were:

1. `MockTransport` was converted so `DispatchAsync(...)` is the real implementation path. It now captures requests, fires `OnRequestStart(...)` first, drives queued responses/errors directly, honors delayed cancellation via `Task.Delay(..., ct)`, and reports missing queued responses through `OnResponseError(...)` instead of throwing. The buffered `SendAsync(...)` helper remains only as a compatibility wrapper over `TransportDispatchHelper.CollectResponseAsync(...)`.
2. `RecordReplayTransport` was converted so `DispatchAsync(...)` is authoritative for all three modes:
   - record mode wraps the downstream handler in a new `RecordingHandler` that buffers response data in `SegmentedBuffer`, records artifacts from live callbacks, and disposes buffered data on success/error/cancellation paths
   - replay mode resolves the recorded entry first, then emits direct callbacks with `OnRequestStart(...)` as the first callback on matched or strict-mismatch synthetic paths
   - passthrough mode delegates straight to the inner transport dispatch path
   The buffered `SendAsync(...)` helper likewise remains as a compatibility collector over dispatch.
3. The test-only `HttpPipeline` compatibility adapter was replaced with `TestInterceptorPipeline`, which now uses the public `TransportDispatchHelper` surface rather than the internal `DispatchBridge`.
4. The remaining middleware-era test files were renamed to interceptor-era names and updated to use `TestInterceptorPipeline` instead of the removed `HttpPipeline` symbol. The old `Tests/Runtime/Pipeline/HttpPipelineTests.cs` coverage was folded into `InterceptorPipelineTests.cs`, then deleted.
5. Added direct dispatch-contract regression tests for `MockTransport` and expanded `RecordReplayTransport` tests so callback ordering is verified at the transport boundary, not only through `UHttpClient`.

## Decisions / Trade-Offs

1. **Buffered `SendAsync(...)` compatibility was retained on test transports:** instead of deleting the public methods outright, both `MockTransport` and `RecordReplayTransport` now treat them as thin buffered collectors over `DispatchAsync(...)`. This keeps existing helper call-sites viable while ensuring the handler-driven path is the single behavior source of truth.
2. **The test pipeline shim stays test-only but no longer mimics removed runtime naming:** `TestInterceptorPipeline` preserves the small `ExecuteAsync(...)` convenience surface for broad test migration without reintroducing the deleted `HttpPipeline` name or leaning on runtime-only compatibility code.
3. **Record mode still batches file persistence behind `SaveRecordings()` / dispose auto-flush:** recording now happens from live callbacks, but the file-writing lifecycle remains the existing queue + flush model instead of forcing an on-every-response disk write.
4. **Replay strict mismatches now follow the transport callback contract in dispatch mode:** synthetic replay failures call `OnRequestStart(...)` first and then `OnResponseError(...)`, rather than surfacing as a raw exception path from the transport wrapper itself.
