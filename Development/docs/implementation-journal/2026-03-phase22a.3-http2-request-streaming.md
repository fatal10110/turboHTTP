# Phase 22a.3 Step 1: HTTP/2 Request-Body Streaming Send Path

**Date:** 2026-03-19  
**Phase:** 22a.3 Step 1  
**Status:** Implemented and validated in a local HTTP/2 runtime harness

## What was implemented

Completed the first HTTP/2 streaming transport slice for Phase 22a.3 by removing the buffered-only request-body restriction from `Http2Connection`:

1. `Http2Connection.DispatchAsync(...)` no longer rejects non-buffered request bodies. It now determines body presence from the 22a request-content model instead of assuming `TryGetBufferedContent(...)` must succeed.
2. `Http2Connection.Send.cs` now has two request send paths:
   - **buffered fast path:** existing `ReadOnlyMemory<byte>` bodies still write DATA frames directly without renting a send buffer
   - **streaming path:** non-buffered bodies open a `RequestBodyReadSession`, rent a pooled send buffer lazily, read incrementally, and pace DATA writes by the existing connection/stream window logic
3. Known-length streaming request bodies now enforce the declared byte count during HTTP/2 send. If the session ends before the declared length is produced, the send path throws instead of silently truncating the body.
4. Unknown-length streaming request bodies now terminate with an explicit zero-length `DATA` frame carrying `END_STREAM` once EOF is observed. This keeps the implementation single-buffered and avoids speculative read-ahead.
5. Send-side failure handling was tightened: if request HEADERS were already sent but local body production fails before the request stream is half-closed, `DispatchAsync(...)` now sends a best-effort `RST_STREAM(CANCEL)` to avoid leaving a half-open request stream on an otherwise-live connection.
6. Added focused HTTP/2 regressions for:
   - known-length `StreamRequestBody` upload
   - unknown-length `FactoryRequestBody` upload with explicit EOF/`END_STREAM`
   - stream-window exhaustion/unblock behavior on a non-buffered request body

## Decisions and trade-offs

1. **Unknown-length HTTP/2 request bodies finish with an empty `DATA+END_STREAM` frame.** This adds at most one extra frame per unknown-length upload, but it avoids either double-buffering or a look-ahead read that would complicate the zero-allocation send path and session ownership model.
2. **The streaming send buffer is rented lazily and only for non-buffered bodies.** Buffered request bodies keep the existing zero-copy `ReadOnlyMemory<byte>` path, so small in-memory uploads do not regress into extra pool traffic.
3. **`Length == 0` non-buffered bodies are treated as header-only requests.** The shared `HasRequestBody(...)` check avoids opening a body session or sending an unnecessary empty DATA frame for zero-length stream/factory bodies with a declared known length.
4. **Best-effort `RST_STREAM(CANCEL)` on local body-production failure is scoped narrowly.** It only runs after HEADERS were already sent and before local end-of-stream was sent, so normal response/handler failure paths are unchanged.
