# Phase 22a.4 Step 3: Direct Buffered Response Collector Path

**Date:** 2026-03-20  
**Phase:** 22a.4 Step 3  
**Status:** Implemented and validated in a local compile/run harness

## What was implemented

Completed the next validated Phase 22a.4 slice by adding the detached buffered-body contract and teaching the buffered collector to transfer already-buffered response ownership instead of draining/copying it:

1. Added the detached buffered-body contract to `IResponseBodySource`.
   - Added new `Runtime/Core/DetachedBufferedBody.cs`
   - added `IResponseBodySource.TryDetachBufferedBody(out DetachedBufferedBody body)`
   - the detached body carries either:
     - a single-segment `ReadOnlyMemory<byte>` with an optional disposal owner, or
     - a multi-segment `ReadOnlySequence<byte>` with an owner that releases the underlying pooled buffers

2. Updated `BufferedResponseCollectorHandler` to use a zero-copy detach fast path.
   - the collector now checks `TryDetachBufferedBody(...)` before `TryGetBufferedData(...)`
   - on detach success it:
     - skips the `ReadAsync` drain loop
     - skips `GetTrailersAsync(...)`
     - skips `DisposeAsync()` on the source
     - constructs `UHttpResponse` directly from the detached sequence/owner
   - fallback behavior for non-detachable sources remains the existing buffered-data copy path or the segmented drain loop

3. Implemented detach support for already-buffered body sources.
   - `BufferedResponseBodySource` now detaches when it still owns plain buffered memory and has not started reading
   - `MockResponseBodySource` now exposes the same behavior for test scenarios
   - `PluginContext.ObservedBodySource` now forwards the detach contract

4. Implemented the real transport-side detach path for HTTP/2 buffered responses.
   - `Http2ResponseBodySource` now detaches only when:
     - the source is terminal/completed
     - trailers are already resolved successfully
     - no consumer has started reading body bytes
   - detaching drains the queued pooled DATA chunks into a detached ownership object without copying payload bytes
   - connection-level buffered-byte accounting is released immediately on detach, and the body-source lifetime is released without waiting for the buffered collector
   - small multi-frame HTTP/2 responses now preserve their queued chunk boundaries as a detached `ReadOnlySequence<byte>` instead of being recopied through `SegmentedBuffer`

5. Added the narrow HTTP/1.1 detach case that is actually valid for the current architecture.
   - `Http11ResponseBodySource` now returns a detachable empty body only when the response is already terminal with no unread content
   - normal socket-backed HTTP/1.1 bodies still return `false` because the body source owns a live reader, not a fully-buffered payload

6. Added focused tests for the new fast path.
   - buffered collector tests now verify that detach:
     - skips `ReadAsync`
     - skips trailer retrieval
     - skips source disposal
     - preserves detached sequence ownership until `UHttpResponse.Dispose()`
   - HTTP/2 now has direct source-level coverage proving:
     - a completed queued body detaches as a multi-segment buffered sequence
     - detach is rejected once a consumer has started reading body bytes

## Decisions and trade-offs

1. **Detach is restricted to ownership shapes that are safe to transfer into `UHttpResponse`.**  
   `DetachedBufferedBody` uses an `IDisposable` owner instead of arbitrary callbacks so the response finalizer can still release pooled buffers if the caller forgets to dispose the response.

2. **`BufferedResponseBodySource` refuses detach when it was created with an `onDispose` callback.**  
   That callback shape is not guaranteed to be finalizer-safe, and the main raw-socket HTTP/1.1 parsed-response callback currently returns pooled parser objects via an object-pool lock. Those cases still fall back to the existing copy path rather than risk leaking or returning pooled state unsafely.

3. **HTTP/2 detach is terminal-only and pre-read only.**  
   The source now tracks whether any body bytes were consumed and rejects detach after partial reads. This prevents re-exposing already-consumed bytes to a second caller.

4. **HTTP/1.1 detach is intentionally minimal in this step.**  
   The normal HTTP/1.1 response path does not hold a fully buffered body inside `Http11ResponseBodySource`; it owns a live socket reader. Only the already-completed empty-body case can be detached without inventing a new ownership model for live transport buffers.
