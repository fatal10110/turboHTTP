# Phase 22a.3: HTTP/2 Streaming Review Fixes

**Date:** 2026-03-20  
**Phase:** 22a.3 review follow-up  
**Status:** Implemented and validated in local HTTP/2 compile/run harnesses

## What was implemented

Closed the issues called out in `Development/docs/phases/phase22a/review-22a.3-http2-streaming.md` across the HTTP/2 streaming receive path, response-body source, and tests:

1. `Http2ResponseBodySource` now has a real `DrainAsync(...)` path.
   - HTTP/2 drain now aborts the stream and awaits cleanup instead of silently returning.
   - faulted bodies still surface their original error instead of being converted into dispose noise.
2. Response-body accounting now matches actual buffer ownership.
   - per-stream `_bufferedBytes` is decremented only when a chunk buffer is returned to `ArrayPool`
   - connection-level buffered-byte accounting now tracks flow-controlled bytes, including padding
   - cleanup releases both unread data bytes and unread flow-controlled bytes so abort/reset paths do not leak connection credit
3. Body-source lifetime and abort races were tightened.
   - cleanup publication is now single-assignment and awaitable for all callers
   - queue completion happens before draining unread buffers so in-flight producer writes cannot strand pooled buffers
   - queue write failure after completion is treated as abort/buffer-full instead of throwing from the read loop
4. Flow-control update ordering was corrected.
   - connection `WINDOW_UPDATE` bookkeeping is now applied before the frame send
   - connection receive-window recovery now targets a larger internal connection window (1 MiB, bounded by the aggregate buffered-byte cap) instead of oscillating only inside the RFC default 65535-byte window
   - stall timeout reads now use atomic/volatile access, and stalled-stream tests use internal `Http2Options` overrides instead of reflection
5. Low-priority cleanup items from the review were also finished.
   - `SingleReaderChannel<Http2ResponseBodyChunk>` now uses item capacity only as a channel implementation detail; byte-level backpressure remains the real limiter
   - stale `link.xml` preservation for `SingleReaderChannel<ReadOnlyMemory<byte>>` was removed
   - `_recentlyResetStreams` now gets opportunistic hard trimming when pathological reset churn exceeds the soft/hard entry thresholds between maintenance passes
6. The HTTP/2 test surface was updated to validate the reviewed behavior rather than the old pre-streaming assumptions.
   - stream-level `WINDOW_UPDATE` is now asserted only after consumer reads
   - added coverage for drain-abort, aggregate connection buffering gate, concurrent streamed responses, per-stream buffer-full reset, and non-reflection stall tuning
   - tests were updated to tolerate protocol-correct ordering where connection `WINDOW_UPDATE` can precede the eventual `RST_STREAM`

## Decisions and trade-offs

1. **HTTP/2 drain remains abort-based rather than read-and-discard.**  
   That preserves the `IResponseBodySource` contract without forcing an unnecessary consume loop on multiplexed HTTP/2 streams where connection reuse is not blocked by leaving a single stream unread.
2. **Connection buffered-byte accounting follows flow-control semantics, not payload-only semantics.**  
   This is slightly stricter because padding bytes are now counted, but it makes the aggregate buffering gate line up with the actual receive window consumed on the wire.
3. **Connection receive-window recovery now uses a transport-internal 1 MiB target.**  
   This fixes the review’s throughput concern without adding new public API yet; the target is still bounded by the internal aggregate buffering cap.
4. **Recently-reset stream tracking uses pragmatic trimming instead of a new queue/ring-buffer type.**  
   The review concern was bounded growth between maintenance passes, not perfect eviction order. Opportunistic trimming only activates under pathological churn and keeps the implementation simple.
