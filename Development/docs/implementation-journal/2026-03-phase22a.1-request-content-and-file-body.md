# Phase 22a.1: Request `Content` Migration and File Body Support

**Date:** 2026-03-17  
**Phase:** 22a.1 follow-up (request model migration slices)  
**Status:** In progress (request body model now reaches `UHttpRequest` and `TurboHTTP.Files`; handler/response/public-client split still pending)

## What Was Implemented

Built the next validated Phase 22a.1 slices on top of the earlier request-body foundations:

1. `UHttpRequest` now stores `Content: UHttpRequestBody` instead of owning raw request bytes directly.
2. Added request builder helpers for:
   - `WithBody(ReadOnlyMemory<byte>)`
   - `WithLeasedBody(IMemoryOwner<byte>, int length)`
   - `WithStreamBody(Stream, long? contentLength, bool leaveOpen)`
   - `WithBodyFactory(Func<CancellationToken, ValueTask<Stream>>, long? contentLength)`
3. Added detached clone behavior through `UHttpRequestBody.CloneDetached()`:
   - buffered/owned bodies clone to detached buffered copies
   - factory bodies clone to a new wrapper over the same factory
   - stream bodies reject detached clone
4. Added internal `CopyWithSharedContent()` on `UHttpRequest` for same-dispatch copy-on-write mutation flows, then migrated the affected interceptors:
   - `AdaptiveInterceptor`
   - `AuthInterceptor`
   - `DefaultHeadersInterceptor`
   - `CookieInterceptor`
   - `DecompressionInterceptor`
5. Updated immediate request-model consumers to avoid the old raw-body assumption:
   - `BackgroundNetworkingInterceptor` now uses `RequestBodyReplayability`
   - `PluginContext` request mutation signature now keys off the `Content` object instead of raw body memory
   - `RedirectInterceptor` now explicitly rejects non-buffered redirect body reuse with a clear `InvalidRequest` error rather than silently misbehaving
6. Added `FileRequestBody` and `WithFileBody(...)` in `TurboHTTP.Files`.

## Decisions / Trade-Offs

1. **Temporary buffered compatibility view remains on `UHttpRequest.Body`:** it now forwards only when `Content` is already buffered and throws for streaming/non-buffered bodies. This keeps the repo moving incrementally without silently pretending a stream body is empty.
2. **Same-dispatch mutations use shared content instead of detached clones:** this prevents header/timeout/auth/default-header rewrites from breaking stream-backed bodies before transport streaming is implemented.
3. **Detached clone stays strict:** `StreamRequestBody` still throws on `Clone()`, matching the Phase 22a distinction between sequential replayability and detached cloneability.
4. **Redirect reuse is explicit, not implicit:** redirect follow-up requests only reuse buffered bodies. Non-buffered bodies now fail with a clear request error until the later transport/body phases make that flow safe.
5. **`FileRequestBody` lives only in `TurboHTTP.Files`:** Core remains file-I/O free.
