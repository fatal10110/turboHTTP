# Phase 22a.5 Step 5: Bounded Observability Capture

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 5  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the observability-specific 22a.5 rewrite so logging, metrics, and monitor capture now respect streaming response lifetimes instead of assuming buffered responses:

1. Added a shared streaming observation wrapper for non-buffered response bodies.
   - `ObservedResponseBodySource` now underpins the streaming observability path
   - logging, metrics, and monitor handlers wrap non-buffered `IResponseBodySource` instances instead of only peeking at `TryGetBufferedData(...)`
   - wrappers observe incremental reads, `DrainAsync(...)`, aborts, trailer fetches, and final dispose-time completion

2. Moved streaming completion/finalization to response-body disposal.
   - `LoggingHandler`, `MetricsHandler`, and `MonitorHandler` no longer finalize streaming responses when `OnResponseStartAsync(...)` returns
   - completion happens when the wrapped body source is disposed, which matches `SendStreamingAsync(...)` semantics
   - per-request completion/capture latches were hardened so the wrappers do not double-log or double-capture when downstream handlers fail

3. Finished bounded streaming request/response capture behavior.
   - `LoggingInterceptor` now emits streaming request-body metadata (`length` + `replayability`) instead of forcing buffering
   - `MonitorInterceptor` records a request-body capture note for non-buffered request bodies and `HttpMonitorEvent.GetRequestBodyAsString()` surfaces that note
   - `MonitorHandler` captures bounded response prefixes incrementally for streaming responses and records the final event only after disposal

4. Wired body-read failures after response start into observability completion.
   - logging now emits an error log on streaming body read/trailer failures instead of a misleading completion log
   - metrics count those completions as failures
   - monitor snapshots classify those failures as transport errors while still retaining the observed response prefix

5. Added actual request-body byte accounting for streaming uploads.
   - `MetricsHandler` consumes `TransportBehaviorFlags.RequestBodyBytesSent` on completion
   - `Http11RequestWriteState` now tracks actual request body bytes written
   - `RawSocketTransport` stores the observed HTTP/1.1 request-body byte count into `RequestContext` even when serialization fails partway through the upload
   - `Http2Connection` now records actual request-body bytes sent from the DATA-frame path without counting protocol framing bytes

## Decisions / trade-offs

1. **Streaming observability remains wrapper-based and protocol-agnostic.**  
   The handlers observe only the `IResponseBodySource` abstraction. They do not branch on HTTP/1.1 vs HTTP/2 behavior, which keeps the module boundary intact and leaves drain/abort semantics inside the transport.

2. **Streaming completion is tied to `DisposeAsync()` rather than EOF alone.**  
   This keeps the logging/metrics/monitor lifecycle aligned with response ownership. Reaching EOF without disposing the response is not considered complete.

3. **Actual upload-byte accounting tracks request-body payload bytes only.**  
   HTTP/1.1 chunk framing bytes and HTTP/2 DATA frame headers are intentionally excluded. Metrics now represent payload bytes sent, not wire framing overhead.

4. **Streaming request bodies still do not get force-buffered for observability.**  
   The monitor/logging UX now shows an explicit metadata marker instead of silently producing an empty body view, but the implementation still avoids opening read sessions or materializing copies just for diagnostics.
