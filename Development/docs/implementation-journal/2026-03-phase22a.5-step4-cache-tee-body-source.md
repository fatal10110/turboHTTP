# Phase 22a.5 Step 4: Cache Tee Body Source

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 4  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the cache-specific 22a.5 rewrite so cache storage no longer depends on whole-body buffering for streaming responses:

1. Added a bounded cache-body policy limit.
   - `CachePolicy` now exposes `MaxCacheableResponseBodyBytes`
   - default is 8 MiB
   - values must be greater than zero

2. Reworked `CacheStoringHandler` into a mixed buffered/streaming store path.
   - already-buffered response bodies keep a buffered fast path and still queue cache writes after the downstream handler returns
   - non-buffered response bodies now install a `TeeBodySource`
   - known-length responses larger than `MaxCacheableResponseBodyBytes` skip tee installation entirely

3. Added `TeeBodySource` for streaming cache accumulation.
   - wraps the underlying `IResponseBodySource`
   - mirrors bytes into a `SegmentedBuffer` as the consumer reads
   - marks cache commit eligibility only after natural EOF
   - discards the accumulator on early dispose/abort instead of producing partial cache entries

4. Enforced bounded accumulation and silent detach behavior.
   - if the running body size exceeds `MaxCacheableResponseBodyBytes`, the accumulator detaches and response delivery continues
   - if the accumulator write path faults, cache capture detaches and response delivery continues
   - cache storage still queues asynchronously through the existing mutation/pending-store machinery

5. Preserved the cache queue ordering semantics for buffered bodies.
   - existing "store starts only after downstream response completion" behavior remains intact for already-buffered responses
   - this keeps the current pending-store coordination tests valid while adding streaming-safe behavior for non-buffered sources

## Decisions / trade-offs

1. **Buffered responses keep a fast path instead of forcing everything through the tee.**  
   This preserves the established queue timing for already-buffered sources and avoids wrapping work when the full response is already in memory.

2. **`TeeBodySource` commits on `DisposeAsync()` after natural EOF, not immediately when `ReadAsync` returns 0.**  
   This keeps the cache write aligned with response-lifetime completion and gives the wrapper one place to enforce the "EOF + trailers" rule before queuing the store.

3. **Trailer fetch failures during cache finalization discard the cache entry instead of surfacing a new dispose-time error.**  
   The wrapper requires successful trailer completion before caching, but it does not introduce new user-visible failures during `DisposeAsync()` just to satisfy cache bookkeeping.

4. **The size cap is applied twice.**  
   A pre-tee known-length check avoids pointless wrapper allocation for obviously too-large bodies, while incremental accounting handles unknown-length streaming responses.
