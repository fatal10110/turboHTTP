## Pass 3 Resolution Summary

1. **Decompression hardening landed:** `DecompressionHandler` now catches `IOException` and `InvalidDataException`, enforces a max decompressed-body size, supports `x-gzip` plus multi-value `Content-Encoding` chains, and preserves unsupported encodings instead of stripping headers it cannot decode.
2. **Retry and redirect safety issues are fixed:** retry telemetry remains correct on terminal success, retryable errors after response commitment no longer loop past an already-committed handler path, and `RedirectHandler` now fails fast if a spawned dispatch completes without any terminal callback.
3. **Observability hot-path allocations were reduced:** `LoggingHandler` allocates its preview buffer lazily, `ReadOnlySequenceStream.Position` is now O(1), and monitor capture no longer duplicates buffered response bodies through `_responseBody?.ToArray()`.
4. **Cache background work is now bounded:** fire-and-forget store/revalidation tasks are explicitly observed, tied to interceptor disposal via cancellation, and stale-while-revalidate retention still preserves the valid stale window.
5. **Clone-ownership and minor allocation issues are fixed:** clone-on-write interceptors now dispose cloned requests on synchronous dispatch throws, redirect request rebuilding now reuses full array-backed bodies where possible, `GracePeriodBeforeQueue` is now consumed, and cookie merging no longer allocates via `Split(';')`.
6. **Targeted regressions were added:** decompression now has truncation/limit/encoding-chain coverage, retry has committed-error coverage, redirect has completion-bridge coverage, metrics verifies 3xx classification, and cache verifies disposal-bound revalidation cancellation.

## Decisions / Trade-Offs

1. **H-4 remains deferred by design:** full streaming decompression is still a larger architectural change than the rest of the Pass 3 fix set. The limitation stays documented rather than being hidden as an untracked known issue.
2. **Redirect body reuse is best-effort:** the fix avoids unconditional copies for full array-backed request bodies without broadening Core request-body ownership APIs during Phase 22.3.
3. **Background cache cancellation favors correctness over late completion:** disposal now cancels outstanding background work instead of allowing revalidation/storage to continue after interceptor teardown.
