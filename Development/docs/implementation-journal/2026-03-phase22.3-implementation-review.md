## Pass 3 Resolution Summary

1. **Decompression hardening landed:** `DecompressionHandler` now catches `IOException` and `InvalidDataException`, enforces a max decompressed-body size, supports `x-gzip` plus multi-value `Content-Encoding` chains, and preserves unsupported encodings instead of stripping headers it cannot decode.
2. **Retry and redirect safety issues are fixed:** retry telemetry remains correct on terminal success, retryable errors after response commitment no longer loop past an already-committed handler path, and `RedirectHandler` now fails fast if a spawned dispatch completes without any terminal callback.
3. **Observability hot-path allocations were reduced:** `LoggingHandler` allocates its preview buffer lazily, `ReadOnlySequenceStream.Position` is now O(1), and monitor capture no longer duplicates buffered response bodies through `_responseBody?.ToArray()`.
4. **Cache background work is now bounded:** fire-and-forget store/revalidation tasks are explicitly observed, tied to interceptor disposal via cancellation, and stale-while-revalidate retention still preserves the valid stale window.
5. **Clone-ownership and minor allocation issues are fixed:** clone-on-write interceptors now dispose cloned requests on synchronous dispatch throws, redirect request rebuilding now reuses full array-backed bodies where possible, `GracePeriodBeforeQueue` is now consumed, and cookie merging no longer allocates via `Split(';')`.
6. **Targeted regressions were added:** decompression now has truncation/limit/encoding-chain coverage, retry has committed-error coverage, redirect has completion-bridge coverage, metrics verifies 3xx classification, and cache verifies disposal-bound revalidation cancellation.

## Pass 5 Resolution Summary

1. **Foreground cache revalidation now restores the original request context in all paths:** `CacheInterceptor.RevalidateAsync` restores `RequestContext.Request` before replaying user-visible callbacks and again in `finally` before disposing the internal conditional request clone.
2. **Decompression now guarantees a terminal error callback for unexpected handler failures:** `DecompressionHandler.OnResponseEnd` keeps the existing IO/data-corruption mapping and now routes any other exception escaping buffered decompression through `_inner.OnResponseError(...)`.
3. **Redirect completion now preserves real downstream terminal failures:** `RedirectHandler` resolves or faults its completion source even when `_inner.OnResponseEnd(...)` / `_inner.OnResponseError(...)` throw synchronously, so redirected requests no longer collapse into the generic completion-bridge failure.
4. **Cache store ownership is now guarded on synchronous queue failure:** `CacheStoringHandler.OnResponseEnd` disposes the detached `SegmentedBuffer` if `QueueStoreResponse(...)` throws before ownership transfers to background mutation work.
5. **Monitor capture is decoupled and bounded:** `MonitorInterceptor.LogCaptureFailure` now uses a dedicated throttle lock, and `MonitorHandler` only retains a bounded response-body prefix sized for final monitor snapshot policies plus binary detection.
6. **New regressions cover the new pass:** cache verifies request-context restoration after failed revalidation, decompression verifies downstream `OnResponseData(...)` failures become `OnResponseError(...)`, and redirect verifies downstream `OnResponseEnd(...)` failures surface as the original exception rather than the generic “terminal callback” error.

## Pass 4 Findings (Historical)

### BLOCKER

| ID | Source | Component | Description |
|----|--------|-----------|-------------|
| P4-B1 | Infra | `CacheInterceptor.RevalidateAsync` (line ~388) | `context.UpdateRequest(conditionalRequest)` is not restored on exception. If `CollectBufferedResponseAsync` throws (transport error, cancellation), context retains the conditional request with `If-None-Match`/`If-Modified-Since` headers. Outer interceptors (logging, metrics, monitor) will report the wrong request for the remainder of pipeline tear-down. Fix: add try/catch around `CollectBufferedResponseAsync` that restores `context.UpdateRequest(request)` before re-throwing — same one-line pattern already used in `AuthInterceptor`. |

### WARNING

| ID | Source | Component | Description |
|----|--------|-----------|-------------|
| P4-W1 | Infra | `DecompressionHandler.OnResponseEnd` (line 90) | Exception filter only catches `InvalidDataException` and `IOException`. Any other exception from `DecompressBufferedBody` (e.g., `OutOfMemoryException`, exception propagating from `_inner.OnResponseData`) escapes without calling `_inner.OnResponseError` or `_inner.OnResponseEnd`, leaving downstream handlers without a terminal callback. Fix: add a broad top-level `catch (Exception)` after the existing filtered catch that calls `_inner.OnResponseError`. |
| P4-W2 | Infra | `RedirectHandler.OnResponseEnd` (lines 80–81, 108–109) | If `_inner.OnResponseEnd(trailers, context)` throws synchronously, `_completion.TrySetResult(null)` is never called. The outer `await redirectHandler.Completion` hangs forever. Fix: wrap `_inner.OnResponseEnd` in try/finally that guarantees `_completion.TrySetResult` or `_completion.TrySetException` is called. |
| P4-W3 | Both | `CacheStoringHandler.OnResponseEnd` (line ~95) | Implicit ownership assumption: if `QueueStoreResponse` itself throws synchronously, `bodyToStore` (a `SegmentedBuffer`) leaks. Low probability (only possible if `baseKey` is null/empty), but the ownership transfer is not guarded. Fix: wrap `QueueStoreResponse` in try/catch that disposes `bodyToStore` on failure. |
| P4-W4 | Infra | Multiple interceptors | Clone-then-dispose pattern in `catch` blocks is fragile — if `next` captured the request reference in an async closure before yielding, the dispose races with the async body. Safe in current code (all interceptors copy locally before yielding), but undocumented assumption. |
| P4-W5 | Infra | `MonitorInterceptor.LogCaptureFailure` (line ~584) | Rate-limit state (`_nextCaptureErrorLogUtcTicks`, `_suppressedCaptureErrorCount`) is guarded by `HistoryLock`, coupling diagnostic throttling to the hot capture path. Under high throughput with frequent capture failures, this serializes all event ingestion. Fix: use a dedicated `_logThrottleLock` or `Interlocked.CompareExchange` on `_nextCaptureErrorLogUtcTicks`. |
| P4-W6 | Infra | `DispatchBridge`, `RedirectHandler` | `TaskContinuationOptions.ExecuteSynchronously` in `BridgeDispatchCompletion` can cause deep synchronous stack nesting across redirect hops. With a default max of 10 hops this is unlikely to overflow, but the nesting is unbounded if the limit is raised. |
| P4-W7 | Network | `RetryDetectorHandler` (line 22) | `OnRequestStart` is forwarded to `_inner` unconditionally, but when a 5xx triggers a retry, the `_inner` handler receives multiple `OnRequestStart` calls without matching terminal callbacks. The current architecture mitigates this (retry uses a detector wrapper, not the real downstream), but custom handlers stacked below retry would see inflated request-start counts. |
| P4-W8 | Network | `MonitorHandler` (lines 35–43) | Accumulates entire response body in `SegmentedBuffer` without size check. `MaxCaptureSizeBytes` is only applied when creating the final snapshot. A large response (file download) could cause significant memory pressure. Fix: short-circuit accumulation once buffer exceeds `MaxCaptureSizeBytes`. |
| P4-W9 | Network | `CookieInterceptor.MergeCookieHeaders` (lines 96–97) | Allocates `HashSet<string>` and `List<string>` per merge operation on every request with both user-supplied and jar-supplied cookies. Acceptable for v1 (not a hot path for most users). |
| P4-W10 | Infra | `CacheInterceptor.GetSignatureSnapshot` (Variants.cs line ~119) | `OrderBy().ToArray()` runs inside `_indexLock`. For typical 1–3 header signatures this is trivial, but for pathological cases (up to `MaxVaryHeaders = 32`) this holds the lock during sort + allocation. |

## Pass 5 Closure Notes

1. **P4-W2 completion faulting was implemented without recursive error delivery:** completion helpers now run outside the redirect-resolution `try/catch`, so downstream terminal failures fault the bridge exactly once.
2. **P4-W8 monitor capping preserves monitor UX semantics:** the handler keeps enough prefix bytes for configured previews plus binary detection while still reporting the true original response size and truncation state.

## Decisions / Trade-Offs

1. **H-4 remains deferred by design:** full streaming decompression is still a larger architectural change than the rest of the Pass 3 fix set. The limitation stays documented rather than being hidden as an untracked known issue.
2. **Redirect body reuse is best-effort:** the fix avoids unconditional copies for full array-backed request bodies without broadening Core request-body ownership APIs during Phase 22.3.
3. **Background cache cancellation favors correctness over late completion:** disposal now cancels outstanding background work instead of allowing revalidation/storage to continue after interceptor teardown.
4. **P4-W4 (clone-dispose fragility) is accepted for v1:** all current interceptors copy the request reference locally before yielding, making the pattern safe. The assumption is implicit but consistently applied.
5. **P4-W7 (retry multiple OnRequestStart) is accepted:** the retry detector wraps the real downstream handler, so only the detector's inner sees duplicates. Custom handlers below retry are not a documented supported pattern. Will document the constraint if custom handler stacking becomes a public API.
6. **P4-W9 (cookie merge allocations) deferred to Phase 24 performance pass:** cookie merging is not a hot path for most users.
7. **P4-W10 (sort inside index lock) is accepted for v1:** typical Vary signatures are 1–3 headers; pathological 32-header case is unlikely in practice.
