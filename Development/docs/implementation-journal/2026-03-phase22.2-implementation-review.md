## Findings

### P0-1: Missing `using System.Collections.Generic` in `Http11ResponseParser.cs` (compile error)

**Source:** Network architect (F-1)
**File:** `Runtime/Transport/Http1/Http11ResponseParser.cs:63`

The new `EnumerateBodySegments()` method returns `IEnumerable<ReadOnlyMemory<byte>>`. The file lacks `using System.Collections.Generic;`. Unity targets .NET Standard 2.1 / C# 8, which does not support implicit usings. **This prevents compilation on all Unity targets** (Editor, Standalone, iOS, Android).

**Fix:** Add `using System.Collections.Generic;` to the using directives.

---

### P0-2: `ParsedResponse` body buffers leaked on pool return (per-response memory leak)

**Source:** Network architect (F-2)
**File:** `Runtime/Transport/Http1/Http11ResponseParser.cs:53-61` (`Reset` method)
**File:** `Runtime/Transport/RawSocketTransport.cs:747-783` (`EmitParsedResponse`)

Before Phase 22.2, body ownership transferred to `UHttpResponse` during `BuildResponse()`. Now `EmitParsedResponse` iterates body segments via `EnumerateBodySegments()` and emits each to the handler, but `Reset()` clears all fields without releasing pooled resources:

- When `BodyFromPool == true`: the `byte[]` rented from `ArrayPool<byte>.Shared` is never returned.
- When `SegmentedBody != null`: the `SegmentedBuffer` with pooled segments is never disposed.

Over the lifetime of a connection pool, this causes unbounded `ArrayPool` growth — particularly severe on mobile platforms.

**Fix:** Add a `ReleaseBodyBuffers()` method to `ParsedResponse` that returns pooled arrays and disposes `SegmentedBody`. Call it before `ParsedResponsePool.Return(parsed)` in `EmitParsedResponse`. Have `Reset()` call `ReleaseBodyBuffers()` as a safety net.

### P0-3: `RecordReplayTransport.Dispose()` throws exceptions (IDisposable contract violation)

**Source:** Infrastructure architect (D-1)
**File:** `Runtime/Testing/RecordReplayTransport.cs:376-408`

`Dispose()` catches exceptions from `SaveRecordings()` and `_innerTransport.Dispose()` but then re-throws them via `throw disposeException;`. This violates the `IDisposable` contract (CA1065, ECMA CLI spec):

- `using` blocks propagate the dispose exception, shadowing any in-flight exception.
- `UHttpClient.Dispose()` may not catch exceptions from transports that re-throw from `Dispose()`.

**Fix:** Replace `throw disposeException;` with `System.Diagnostics.Debug.WriteLine($"[TurboHTTP] RecordReplayTransport dispose error: {disposeException}");`.

### P1-1: Buffered collection no longer preserves cancellation exceptions

**Source:** Round 1 review
**Files:** `Runtime/Core/Pipeline/ResponseCollectorHandler.cs`, `Runtime/Core/Pipeline/DispatchBridge.cs`

`ResponseCollectorHandler.Fail()` now wraps every non-`UHttpException` in `UHttpException(Unknown)`. That breaks the buffered dispatch bridge because `DispatchBridge.CollectResponseAsync()` routes synchronous throws through `Fail()`, and `AttachCompletion()` also routes derived `OperationCanceledException` subtypes there.

As a result:

- synchronous cancellation is no longer preserved as cancellation
- typed cancellation such as `BackgroundRequestQueuedException` is collapsed into `UHttpException(Unknown)`
- buffered callers observe the wrong failure shape and lose cancellation metadata

**Impact:** Regresses the current interceptor/background-networking contract and can break callers/tests that rely on typed cancellation behavior.

**Fix:** Preserve `OperationCanceledException` in `Fail()` instead of wrapping it, matching the pre-22.2 collector contract for both plain and derived cancellation exceptions.

### P1-2: HTTP/2 per-stream handler failures can tear down the whole connection

**Source:** Round 1 review
**Files:** `Runtime/Transport/Http2/Http2Stream.cs`, `Runtime/Transport/Http2/Http2Connection.ReadLoop.cs`

The new direct `IHttpHandler` callbacks in `Http2Stream` run on the shared HTTP/2 read-loop thread. If `OnResponseStart`, `OnResponseData`, or `OnResponseEnd` throws for one stream, that exception escapes the stream callback path and bubbles into `ReadLoopAsync()`.

`ReadLoopAsync()` currently treats unexpected exceptions as connection-fatal and calls `FailAllStreams(...)`, which means one misbehaving interceptor/handler can abort unrelated in-flight streams on the same multiplexed connection.

**Impact:** Violates HTTP/2 stream isolation expectations and turns a single-stream handler bug into a whole-connection failure.

**Fix:** Catch handler exceptions at the stream boundary, fail only the affected stream, and avoid letting per-stream callback faults escape as connection-level read-loop failures.

### P2-1: Post-header HTTP/2 failures skip `RequestFailed` timeline recording

**Source:** Round 1 review
**File:** `Runtime/Transport/Http2/Http2Connection.cs`

When `Http2Connection.DispatchAsync()` catches an exception after response headers have already been received, it calls `handler.OnResponseError(...)` and returns. Unlike the HTTP/1.1 path, it does not record `context.RecordEvent("RequestFailed")` first.

**Impact:** HTTP/2 mid-response failures bypass request-timeline/observability accounting, creating inconsistent diagnostics versus HTTP/1.1.

**Fix:** Record `RequestFailed` before invoking `OnResponseError(...)` in the post-header HTTP/2 error path.

---

### P2-2: Server `RST_STREAM(CANCEL)` misclassified as client-initiated cancellation

**Source:** Network architect (F-8)
**File:** `Runtime/Transport/Http2/Http2Connection.ReadLoop.cs:694`

When the server sends `RST_STREAM` with error code `CANCEL` (0x8), the read loop calls `stream.Cancel()`, creating an `OperationCanceledException(CancellationToken.None)`. A server-initiated `RST_STREAM(CANCEL)` is semantically different from client cancellation — it means the server decided to abort the stream.

Upstream code that checks `OperationCanceledException` to decide whether to retry (e.g., `RetryMiddleware`) will incorrectly treat a server-initiated cancel as a non-retryable client cancellation rather than as a retryable network event.

**Fix:** Treat server `RST_STREAM(CANCEL)` the same as other RST_STREAM error codes — use `stream.Fail(new UHttpException(...))` instead of `stream.Cancel()`.

---

### P2-3: Missing `RST_STREAM` when removing a disposed stream from the read loop

**Source:** Network architect (F-4)
**File:** `Runtime/Transport/Http2/Http2Connection.ReadLoop.cs:196-199`

When `stream.AppendResponseData` throws `ObjectDisposedException` (stream concurrently disposed by cancellation), the code removes the stream from `_activeStreams` but sends no `RST_STREAM`. Per RFC 9113 Section 8.1, the client should send `RST_STREAM` to inform the server. Without this, the server continues sending DATA frames — wasting server bandwidth, client CPU, and connection-level flow control window capacity.

Compare with the handler fault path (lines 202-206) which correctly sends `RST_STREAM`.

**Fix:** Add `_ = SendRstStreamAsync(frame.StreamId, Http2ErrorCode.Cancel);` before the `return` at line 198.

---

### P3-1: `Http2Stream.Complete()`/`Fail()` lack defensive guard against double-set

**Source:** Infrastructure architect (T-1)
**File:** `Runtime/Transport/Http2/Http2Stream.cs:228-247`

`Complete()` and `Fail()` call `_completionSource.SetResult`/`SetException` without guarding against `InvalidOperationException`. The `_responseCompleted` interlocked exchange provides mutual exclusion between `Complete`/`Fail`/`Cancel`, making double-set unreachable under the current single-read-loop design. However, the existing `HandleSettingsFrameAsync` already uses a defensive try/catch around `SetResult` — the same pattern should be applied here for future-proofing.

**Fix:** Wrap `_completionSource.SetResult`/`SetException` calls in `try/catch (InvalidOperationException)` with a comment explaining the guard.

---

### P3-2: `HandleResponseHeaders` returns `true` when stream is disposed

**Source:** Network architect (F-5)
**File:** `Runtime/Transport/Http2/Http2Stream.cs:183`

When a stream has been disposed, `HandleResponseHeaders` returns `true` (success) without setting `HeadersReceived` or invoking the handler. This is misleading — `AppendResponseData` and `Complete` return `false` for no-op paths on disposed streams.

**Fix:** Return `false` when `_disposed != 0`, consistent with other stream methods.

## Pass Areas (both reviewers agree)

- **Architecture & module dependencies:** No violations. `NullHandler` correctly in Transport, `HandlerCallbackException` correctly `internal sealed` in Core, `TransportDispatchHelper` correctly `public static` in Core.
- **Cancellation contract structure:** Consistent `OperationCanceledException` / `OnResponseError` split across transports (pending P1-1 fix for buffered collection path).
- **Handler callback ordering:** Correct across all transports: `OnRequestStart` → `OnResponseStart` → `OnResponseData*` → `OnResponseEnd`.
- **IL2CPP/AOT compatibility:** All patterns safe (`Volatile.Read/Write`, `Interlocked`, `ManualResetValueTaskSourceCore<VoidResult>`, static lambdas, `ArrayPool<ResponseEventSignature>`).
- **Http2Stream pool lifecycle:** Ownership transfer correct; `PrepareForPool()`/`Dispose()` non-reentrancy handled via `Interlocked.CompareExchange`.
- **DispatchBridge.AttachCompletion:** `ExecuteSynchronously` micro-optimization is safe — runs on thread-pool thread, `TaskCompletionSource` uses `RunContinuationsAsynchronously`.
- **Old bridge state keys:** Fully purged (`DeliverResponse`, `SetCancellationException`, `GetCancellationException`, `ResponseErrorStateKey`, `CancellationExceptionStateKey`).
- **Transport body ownership:** `request.DisposeBodyOwner()` fully removed from transport layer.
- **MockTransport/RecordReplayTransport:** Handler emission + response disposal ordering correct per `IHttpHandler` span contract.
