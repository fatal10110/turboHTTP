# Phase 22a.2 Step 1: HTTP/1.1 Known-Length Request Streaming

**Date:** 2026-03-18  
**Phase:** 22a.2 Step 1  
**Status:** Implemented and validated in a runtime harness; Unity/package validation still pending

## What Was Implemented

Completed the first HTTP/1.1 transport slice for Phase 22a.2:

1. `Http11RequestSerializer` now supports non-buffered **known-length** request bodies by opening a `RequestBodyReadSession` and streaming bytes directly to the socket with a pooled 32 KB transfer buffer.
2. The serializer now treats HTTP/1.1 framing headers as **transport-owned** for this path:
   - known-length bodies emit `Content-Length`
   - empty bodies strip both `Content-Length` and `Transfer-Encoding`
   - user-provided `Content-Length` / `Transfer-Encoding` values no longer leak to the wire for these cases
3. Buffered request bodies keep their direct fast path with no extra `Stream` adapter or transfer buffer.
4. `RawSocketTransport` stale-connection retry logic is now conservative for streaming uploads:
   - retries still require an idempotent method
   - retries are blocked for `RequestBodyReplayability.NonReplayable`
5. Added focused serializer and raw-socket transport tests covering:
   - known-length factory-backed upload serialization
   - framing-header normalization
   - end-to-end HTTP/1.1 upload of a non-buffered known-length body
   - no retry for stale reused connections with a non-replayable stream body

## Decisions / Trade-Offs

1. **Unknown-length request bodies still fail fast in Step 1.** `Transfer-Encoding: chunked` upload support remains Step 2 work; the serializer now throws a clear `InvalidOperationException` instead of pretending unknown-length bodies are supported.
2. **Retry logic is intentionally conservative for now.** The new replayability gate prevents invalid retries for one-shot bodies, even though a later 22a.2 step can relax this with explicit "request-body bytes committed" tracking.
3. **Empty-body framing follows the 22a plan, not the old Phase 3 serializer behavior.** User-supplied `Transfer-Encoding` / `Content-Length` headers are stripped instead of preserved when there is no request body.
4. **No new assembly dependencies were introduced.** The work stays inside the existing `TurboHTTP.Transport` -> `TurboHTTP.Core` boundary.
