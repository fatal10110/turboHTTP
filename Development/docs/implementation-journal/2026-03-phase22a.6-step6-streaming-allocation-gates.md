## What was implemented

Added the first repo-owned 22a.6 validation slice for steady-state streaming allocation behavior:

1. Added `Tests/Runtime/Performance/StreamingAllocationGateTests.cs`.
   - `Http11StreamingRead_SteadyStateManagedBytesPerRead_StaysWithinBudget()` measures direct `Http11ResponseBodySource` reads over a deterministic content-length payload.
   - `Http2StreamingRead_SteadyStateManagedBytesPerRead_StaysWithinBudget()` measures direct `Http2ResponseBodySource` reads over pre-enqueued chunk buffers.
   - `DecompressionStreamingRead_KnownAllocationPoint_StaysWithinBudget()` measures the public streaming decompression path through `DecompressionInterceptor` and a callback transport.

2. Kept the measurements scoped to body-read loops rather than request setup.
   - each scenario performs a warmup run first
   - setup allocations happen before the measurement counter is sampled
   - the measured loop only reads streaming bytes until EOF

3. Added a dual-counter strategy so the gates stay useful across Unity/runtime environments.
   - when available, the tests use `GC.GetAllocatedBytesForCurrentThread` via reflection for precise per-thread allocation measurements
   - when that API is unavailable, the tests fall back to `GC.GetTotalMemory(forceFullCollection: true)` with looser budgets because retained-heap deltas are noisier than true allocation counters

4. Documented Phase 22a status in `CLAUDE.md`.

## Decisions / trade-offs

1. **The gates measure steady-state body reads, not whole-request overhead.**  
   The 22a.6 requirement is about per-chunk streaming behavior. Using direct body sources for HTTP/1.1 and HTTP/2 keeps the signal focused on the hot read path instead of client/request setup allocations.

2. **HTTP/2 flow-control side effects were intentionally suppressed for the allocation gate.**  
   The direct `Http2ResponseBodySource` test enqueues chunks with `flowControlledLength = 0` so the measurement does not include `WINDOW_UPDATE` bookkeeping or stream-write effects. Those behaviors are already covered in the existing HTTP/2 protocol tests; this gate is specifically about read-path allocation stability.

3. **Decompression remains a bounded-budget gate, not a zero-allocation gate.**  
   The phase doc already calls out `GZipStream.ReadAsync(...)` as a known allocation point. This test therefore enforces a conservative upper bound instead of pretending the decompression wrapper is zero-allocation today.

4. **Reflection was used only for optional diagnostics and public-path extraction.**  
   The allocation counter reflection is test-only and avoids a hard compile-time dependency on runtime-specific GC APIs. The `UHttpStreamingResponse` body-source extraction mirrors the existing test pattern already used elsewhere in the phase.
