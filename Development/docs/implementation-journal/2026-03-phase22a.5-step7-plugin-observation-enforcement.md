# Phase 22a.5 Step 7: Plugin Observation Enforcement

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 7  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the final 22a.5 slice for plugin capability enforcement so `CapabilityEnforcedInterceptor` and `ObservedHandler` now distinguish read-only observation from real response mutation under the new `IResponseBodySource` model:

1. Tightened request mutation signatures for the new request-body hierarchy.
   - `RequestMutationSignature` now records a stable content hash instead of relying only on object identity
   - buffered/owned request-body variants (`BufferedRequestBody`, `OwnedMemoryRequestBody`) now expose deterministic hash codes based on their current bytes
   - non-replayable request-body variants (`StreamRequestBody`, `FactoryRequestBody`, `FileRequestBody`) intentionally keep identity-based hashes so capability checks do not try to read streams

2. Reworked response observation signatures around body state, not wrapper identity.
   - `ObservedHandler` now wraps the forwarded body in an `ObservedBodySource` and records the response-start signature against that observed body
   - `ObservedBodySource` tracks whether the plugin only inspected metadata or actually read bytes, detached buffered ownership, fetched trailers, aborted, or disposed the body
   - `ResponseEventSignature.ForResponseStart(...)` now includes the underlying body type plus the observation-state hash, allowing read-only monitoring plugins to inspect metadata without being blamed for mutation while still rejecting body wrapping/reading/trailer access without `MutateResponses`

3. Preserved capability enforcement for buffered callers by fixing late-fault completion.
   - Step 7 validation exposed that buffered collection still completed the public response task before the outer dispatch task finished
   - `BufferedResponseCollectorHandler` now stages buffered responses internally and only resolves the public task after dispatch completes successfully
   - late redispatch faults, callback failures, or cancellation now dispose the staged response and fault the caller instead of being hidden behind an already-returned `UHttpResponse`

4. Added targeted regression coverage for the new enforcement rules.
   - read-only monitoring plugins can observe streaming response metadata
   - wrapping a streaming body, reading bytes, or fetching trailers without `MutateResponses` now fails deterministically
   - redispatch without `AllowRedispatch` now surfaces correctly even on the buffered bridge path
   - a direct buffered-dispatch regression now proves late faults after a buffered response has been staged still fail the caller

## Decisions / trade-offs

1. **Observation is modeled as body-state transition, not as wrapper replacement.**  
   Comparing wrapper references would reject harmless metadata-only monitoring. Tracking actual reads, detach, trailer fetch, abort, and disposal keeps the guard aligned with the real mutation boundary.

2. **Request-body hashing stays bounded and stream-safe.**  
   Buffered payloads now participate in mutation detection through stable hashes, while stream/file/factory bodies remain identity-based so capability checks never consume or rewind live uploads.

3. **Buffered response success now depends on dispatch completion, not only body collection.**  
   This keeps plugin redispatch faults and other late failures observable to callers, at the cost of holding the buffered response staged until the dispatch task finishes.

4. **The Step 7 validation fix stayed in Core instead of special-casing plugins.**  
   The masked redispatch fault was a generic buffered-bridge correctness issue, not a plugin-only problem, so the fix belongs in `BufferedResponseCollectorHandler`.
