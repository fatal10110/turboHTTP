# Phase 22a.4 Step 1: `StreamingOptions` Configuration Plumbing

**Date:** 2026-03-20  
**Phase:** 22a.4 Step 1  
**Status:** Implemented and validated in a local compile/run harness

## What was implemented

Completed the first Phase 22a.4 slice by replacing the remaining hardcoded transport streaming thresholds with a runtime-configurable `StreamingOptions` snapshot on `UHttpClientOptions`:

1. Added `Runtime/Core/StreamingOptions.cs` and wired `UHttpClientOptions.Streaming`.
   - Added the Phase 22a.4 Step 1 threshold set:
     - `SmallBufferedRequestThresholdBytes`
     - `DefaultStreamingSendBufferBytes`
     - `DefaultStreamingReceiveBufferBytes`
     - `DefaultHttp2PerStreamReceiveBufferBytes`
     - `BufferedDrainReuseThresholdBytes`
     - `MaxConnectionBufferedBytes`
     - `Http2StallTimeoutSeconds`
   - Added `Clone()` and `IsDefault()` so clients keep snapshot semantics and can still reuse the singleton default transport only when all advanced transport knobs are default.

2. Extended transport creation to preserve streaming configuration without breaking the existing factory API.
   - `UHttpClient` now allocates an owned transport when `StreamingOptions` differ from defaults.
   - `HttpTransportFactory` now has additive overloads for `Register(...)` and `CreateWithOptions(...)` that accept `StreamingOptions`.
   - The previous three-argument advanced-factory path remains intact for compatibility.

3. Plumbed `StreamingOptions` into `RawSocketTransport` and its HTTP/1.1 / HTTP/2 subpaths.
   - `RawSocketTransport` now snapshots streaming options and passes them into:
     - `Http11RequestSerializer`
     - `Http11ResponseBodySource`
     - `Http2ConnectionManager` / `Http2Connection`
   - Proxy-forwarded and tunneled HTTP/1.1 paths now reuse the same streaming thresholds as direct requests.

4. Replaced the existing internal streaming constants where 22a.4 Step 1 already defines runtime-configurable behavior.
   - HTTP/1.1 request streaming send buffers now use `DefaultStreamingSendBufferBytes`.
   - HTTP/1.1 early-dispose/drain behavior now uses `DefaultStreamingReceiveBufferBytes` and `BufferedDrainReuseThresholdBytes`.
   - HTTP/2 request streaming send buffers now use `DefaultStreamingSendBufferBytes`.
   - HTTP/2 per-stream receive buffering now uses `DefaultHttp2PerStreamReceiveBufferBytes`.
   - HTTP/2 aggregate connection buffering now uses `MaxConnectionBufferedBytes`.
   - HTTP/2 stall detection now uses `Http2StallTimeoutSeconds`.
   - The recently-reset stream retention horizon now follows the configured stall timeout instead of the removed fixed timeout constant.

5. Added focused validation tests for the new configuration surface.
   - `UHttpClientOptions.Clone()` now has explicit streaming-option deep-copy coverage.
   - `UHttpClient` now has coverage for streaming-option transport-factory plumbing.
   - Connection option tests now verify that non-default streaming options allocate a dedicated transport instead of reusing the global default.
   - `Http2Connection` now has direct constructor-level coverage proving the configured send buffer, per-stream receive buffer, aggregate buffering cap, and stall timeout are applied.

## Decisions and trade-offs

1. **Transport-factory support was extended additively, not replaced.**  
   `HttpTransportFactory.Register(...)` and `CreateWithOptions(...)` gained new overloads that accept `StreamingOptions`, while the older advanced overloads remain usable. This keeps existing tests and external factory integrations source-compatible.

2. **`RawSocketTransport` snapshots `StreamingOptions` once.**  
   The transport clones the provided `StreamingOptions` and passes that immutable snapshot through to HTTP/1.1 and HTTP/2 internals. This matches the existing `UHttpClientOptions` snapshot model and avoids mutable cross-request configuration drift.

3. **Step 1 only touched thresholds that already existed as internal constants.**  
   The new `StreamingOptions` type exposes the full Step 1 surface now, but `SmallBufferedRequestThresholdBytes` and the future buffered fast-path detach contract remain for later 22a.4 steps. This keeps Step 1 focused on configuration plumbing rather than mixing it with Step 2 or Step 3 behavior changes.

4. **The recently-reset cleanup horizon now follows the configured stall timeout.**  
   This was the narrowest correct replacement for the removed fixed HTTP/2 timeout constant and keeps the retention window aligned with the same stream-lifecycle timeout budget already used elsewhere in the connection maintenance loop.
