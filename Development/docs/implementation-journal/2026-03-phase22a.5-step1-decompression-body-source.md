# Phase 22a.5 Step 1: Streaming Decompression Body Source

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 1  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the first 22a.5 slice by replacing the old buffer-then-decompress middleware path with a mixed buffered/streaming decompression implementation:

1. Added a non-buffered `DecompressionBodySource` path in `Runtime/Middleware/DecompressionHandler.cs`.
   - wraps an `IResponseBodySource`
   - adds an internal ~8 KB read-ahead `BufferedStream`
   - decompresses incrementally via `GZipStream` / `DeflateStream`
   - enforces the decompressed-size limit during reads
   - drains through the decompression stream before exposing trailers
   - aborts the underlying source on decompression faults

2. Preserved a dedicated buffered fast path for already-buffered compressed responses.
   - if the transport already exposes `TryGetBufferedData(...)`, the handler still decompresses immediately into a `BufferedResponseBodySource`
   - restored explicit single-gzip trailer validation on that buffered path so truncated buffered gzip payloads still fail deterministically

3. Kept the handler-layer contract stable.
   - `DecompressionHandler` still owns header rewriting (`Content-Encoding` / `Content-Length` removal)
   - oversize compressed bodies are rejected before handing a response body to downstream handlers
   - downstream callback failures now abort the wrapped decompression source instead of leaking it

4. Updated defaults and tests.
   - `DecompressionInterceptor.DefaultMaxDecompressedBodySizeBytes` now matches the phase plan target: `256 MB`
   - added non-buffered source coverage for both buffered collection and `SendStreamingAsync(...)`
   - updated the downstream handler-failure test to exercise the new body-source model
