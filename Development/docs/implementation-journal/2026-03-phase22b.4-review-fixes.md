# Phase 22b.4 Review Fixes

**Date:** 2026-03-22  
**Phase:** 22b.4 request trailers  
**Status:** Review findings addressed and revalidated

## What was implemented

Completed the follow-up pass for every issue recorded in `phase-22b.4-review.md`.

1. Tightened Core trailer state visibility and builder behavior.
   - `UHttpRequest` now tracks managed `Trailer` header state with an `int` plus `Volatile`/`Interlocked` semantics
   - `WithHeader(...)` now re-synchronizes the managed `Trailer` declaration only when the `Trailer` header itself is touched
   - declared trailer names now reject duplicates and document the Core-vs-Transport prohibited-name split

2. Tightened `UHttpRequestBody` trailer metadata ownership.
   - trailer provider and declared-name references now use volatile reads/writes
   - trailer configuration copy is now assembly-internal and preserves visibility guarantees when cloning detached bodies

3. Fixed HTTP/1.1 empty-body trailer framing.
   - empty request bodies with a trailer provider now resolve to chunked mode instead of `None`
   - zero-length request bodies can now emit the terminal chunk and trailers instead of silently dropping the trailer provider

4. Fixed HTTP/2 managed trailer declaration and allocation behavior.
   - initial HEADERS now skip any user `Trailer` header when request trailers are active and regenerate the declaration from `DeclaredTrailerNames`
   - request-trailer preparation now populates `_headerListScratch` under `_writeLock` and encodes directly from that shared scratch list
   - trailing HEADERS send path now uses the encoded header block directly instead of building a per-request `List<(string,string)>`

5. Closed the review’s minor cleanup items.
   - transport-side `IsRfc9110TChar` validation now lives in `Transport.Internal.EncodingHelper`
   - added the conservative prohibited-request-trailer comment in `TrailerFieldValidator`
   - strengthened the Core trailer-header test to assert a single managed `Trailer` value
   - replaced the timing-dependent HTTP/2 absence assertion with deterministic `END_STREAM` verification on the final DATA frame

## Decisions and trade-offs

1. Kept prohibited trailer-name enforcement split across layers.
   - Core now documents the limitation and validates syntax plus duplicates only
   - Transport remains the enforcement point for the conservative prohibited-name set because Core cannot depend on Transport

2. Used `_headerListScratch` under `_writeLock` for HTTP/2 trailer preparation.
   - this removes the per-request list allocation without changing the existing serialized HEADERS write discipline
   - trailer providers are still evaluated only on trailer-enabled requests, so the no-trailer hot path remains unchanged

3. Preserved the current HTTP/2 end-of-body behavior when no actual trailers remain.
   - final DATA carries `END_STREAM` when the provider returns null/empty or only prohibited trailer fields survive
   - trailing HEADERS carry `END_STREAM` only when actual post-filter trailers remain
