# Phase 22a.1: Request Body Foundations

**Date:** 2026-03-17  
**Phase:** 22a.1 (core body model and public API split)  
**Status:** In progress (foundation slice implemented; request/response API migration and transport adoption still pending)

## What Was Implemented

Started Phase 22a.1 with the first low-risk foundation slice in `TurboHTTP.Core`:

1. Added `RequestBodyReplayability` to make replay semantics explicit before retry/redirect logic is migrated.
2. Added `UHttpRequestBody` as the new request-body abstraction with concrete `EmptyRequestBody`, `BufferedRequestBody`, `OwnedMemoryRequestBody`, `StreamRequestBody`, and `FactoryRequestBody` implementations.
3. Added `RequestBodyReadSession` in `TurboHTTP.Core.Internal` to provide exclusive per-open reader ownership with deterministic disposal.
4. Added focused runtime tests covering replayability, active-reader enforcement, owner disposal, start-position rewind for seekable streams, and disposed-session behavior.

This slice intentionally does **not** migrate `UHttpRequest`, `UHttpClient`, handlers, or transports yet. The repo still uses the old `request.Body` surface until the next step.

## Decisions / Trade-Offs

1. **Foundation-first migration:** this step adds the new body model without yet replacing `UHttpRequest.Body`. That keeps the repo buildable while the clean-break API migration proceeds in smaller validated increments.
2. **Single-reader enforcement lives in the body base type:** `UHttpRequestBody` uses interlocked state to reject concurrent read sessions and to permanently reject reopen on non-replayable bodies.
3. **Dispose defers cleanup until the active session closes:** body disposal now marks the instance as disposed immediately, but owned resources are only released after the in-flight `RequestBodyReadSession` ends. This avoids tearing down a stream or memory owner underneath an active read.
4. **Seekable stream replay uses captured construction position:** `StreamRequestBody` rewinds to the stream position captured at construction time, not to zero.
5. **No transport or public API changes yet:** `IResponseBodySource`, handler contract changes, `SendBufferedAsync` / `SendStreamingAsync`, and `UHttpRequest.Content` migration remain the next 22a.1 steps.
