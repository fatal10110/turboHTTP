**Date:** 2026-03-09

## What Was Fixed

This pass addressed the open findings from `2026-03-phase22.2-implementation-review.md`:

1. Restored buffered cancellation preservation in `ResponseCollectorHandler.Fail()` so synchronous and derived `OperationCanceledException` paths are no longer collapsed into `UHttpException(Unknown)`.
2. Isolated HTTP/2 handler callback faults to the offending stream by catching `OnResponseStart` / `OnResponseData` / `OnResponseEnd` exceptions inside `Http2Stream` and preventing them from escaping the shared read loop as connection-fatal failures.
3. Restored `RequestFailed` timeline recording for post-header HTTP/2 failures before `OnResponseError(...)` is invoked.
4. Added a small internal `HandlerCallbackException` marker so transport code can distinguish handler faults from transport faults and avoid recursively reporting a handler bug back into the same handler via `OnResponseError`.
5. Fixed the HTTP/1.1 parser/transport pooled-body ownership handoff so parsed response buffers are released before pooled `ParsedResponse` instances are returned.
6. Added the missing `System.Collections.Generic` import required by `ParsedResponse.EnumerateBodySegments()`.
7. Changed `RecordReplayTransport.Dispose()` to log and swallow dispose-time failures instead of throwing from `IDisposable`.
8. Aligned HTTP/2 server `RST_STREAM(CANCEL)` handling with the transport error model by surfacing it as `UHttpException(NetworkError)` rather than `OperationCanceledException`.
9. Added the missing best-effort `RST_STREAM(CANCEL)` send when the read loop retires a stream after concurrent body disposal, and tightened stream completion methods with defensive double-set guards.

## Decisions / Trade-Offs

1. **Handler bugs are not transport errors:** the transport now treats handler callback exceptions as caller faults and rethrows them rather than converting them into `UHttpException` transport failures.
2. **HTTP/2 stream isolation takes precedence over aggressive connection failure:** per-stream callback faults now terminate only that stream, with a best-effort `RST_STREAM(CANCEL)` when the peer may continue sending data.
3. **Collector cancellation preservation remains broader than plain task-cancel semantics:** derived `OperationCanceledException` types are preserved as exceptions, while plain canceled tasks still surface through the existing `Cancel()` path.
