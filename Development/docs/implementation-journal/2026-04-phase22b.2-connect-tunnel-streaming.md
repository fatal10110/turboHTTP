# Phase 22b.2 CONNECT Tunnel Streaming and Proxy Retry Parity

**Date:** 2026-04-05  
**Phase:** 22b.2 Streaming Through Proxy Connections  
**Status:** Implemented, review fixes applied, and validated in focused local harnesses

## What was implemented

Completed the remaining HTTPS CONNECT-tunnel portion of Phase 22b.2 and closed the proxy stale-retry gap that became user-visible once proxy connections and tunnels could be returned to the pool.

1. Added CONNECT-tunnel pool rebinding support in `TcpConnectionPool`.
   - `GetConnectionAsync(...)` now accepts an optional pool-key override so a single physical proxy endpoint can keep semaphore accounting on `(proxyHost, proxyPort)` while reusing idle connections by a tunnel-specific identity
   - `PooledConnection` now tracks a mutable `PoolKey` and can be rebound to the tunneled target stream/metadata after CONNECT + TLS setup
   - returned idle connections now enqueue against the active transport binding instead of always using the original physical endpoint key

2. Switched HTTPS proxy dispatch to the lease-based streaming path.
   - CONNECT-tunneled requests now go through `DispatchOnLeaseAsync(...)` instead of the old stream-only path
   - the response body source owns the proxy lease lifetime for streaming responses, so early dispose and drain-or-close reuse rules now match direct HTTP/1.1 behavior
   - same-origin requests through the same proxy can reuse a live CONNECT tunnel; different origins are isolated by a composite tunnel pool key

3. Added proxy stale-retry parity for HTTP/1.1 proxy requests.
   - both plain forward-proxy requests and HTTPS CONNECT-tunneled requests now retry one stale reused connection on `IOException` / `SocketException` when the original request is replayable
   - non-idempotent / non-replayable requests still surface the transport failure without retry, matching the direct HTTP/1.1 path
   - the proxy retry path now records `TransportRetryStale` followed by `TransportProxyConnecting` for consistent observability

4. Hardened proxy lease preparation cleanup.
   - during review, `AcquirePreparedProxyLeaseAsync(...)` was found to leak a lease if CONNECT setup failed after acquiring the proxy connection but before returning it to the caller
   - the helper now disposes the acquired lease on all preparation failures before rethrowing

5. Added focused transport tests for the new behavior.
   - `TcpConnectionPoolTests` now cover tunnel-specific pool-key reuse and confirm per-proxy semaphore limits still apply across different tunnel identities
   - `RawSocketTransportTests` now cover stale forward-proxy retry for idempotent requests and no-retry behavior for non-idempotent requests
   - `RawSocketTransportProxyTunnelTests` cover same-origin tunnel reuse after early streaming-response disposal and ensure different origins do not reuse the same proxy tunnel

6. Applied the Phase 22b.2 review fixes from `phase-22b.2-review.md`.
   - fixed the forward-proxy `Proxy-Authorization` leak when credentials exist but plaintext proxy auth is disallowed
   - switched CONNECT request connection management to the standard `Connection: keep-alive` header
   - corrected the CONNECT reuse invariant so a pooled tunnel is reusable only when it is both already secure and bound to the expected composite tunnel key
   - moved the pool-test accept-count helper out of the integration-only compile guard
   - aligned pool-key tests with the production tunnel-key format
   - fixed the tunnel test proxy relay race and added explicit tunnel stale-retry / over-budget early-dispose coverage

## Decisions and trade-offs

1. Kept tunnel reuse keyed by a composite `(proxy endpoint + origin endpoint)` identity.
   - this prevents incorrect reuse across different CONNECT destinations while preserving the existing per-physical-endpoint semaphore model
   - concurrency remains bounded by the proxy socket endpoint, not by the tunneled origin identities

2. Reused the existing HTTP/1.1 stale-retry rules instead of inventing proxy-specific semantics.
   - proxy reuse is still connection reuse, so the same replayability and body-commit rules apply
   - this keeps direct and proxy paths behaviorally aligned for users and for diagnostics

3. Tunnel reuse needs both the composite key and an established secure tunnel.
   - the review suggested replacing the old `IsSecure`-only guard with a pool-key check
   - in practice, fresh proxy sockets are also rebound under the composite tunnel key before CONNECT, so the correct invariant is `IsSecure && PoolKey == expectedTunnelKey`

4. Left HTTP/2-over-CONNECT out of scope.
   - CONNECT setup still forces HTTP/1.1 ALPN on the inner TLS handshake in this path
   - the phase overview already defers proxy ALPN / HTTP/2 tunnel work to Phase 22c

5. Made the new CONNECT runtime tests self-skipping when the runtime cannot generate a localhost self-signed certificate.
   - local validation showed that certificate generation can fail on some macOS runtimes
   - skipping in unsupported environments is preferable to leaving a platform-specific red test in the default suite

## Specialist rubric pass

### Infrastructure review

- Module boundaries remain unchanged: all runtime changes stay inside `TurboHTTP.Transport`
- No new Unity-engine dependencies were introduced into transport or pool code
- Lease ownership and disposal were re-reviewed after the CONNECT changes; the only issue found was the proxy-lease-preparation leak, which was fixed in this step
- Pool rebinding is only performed while the lease has exclusive ownership of the connection, so the mutable transport binding does not introduce a new shared-state race
- `UpdateTransportBinding(...)` and the CONNECT rebind path are now explicitly documented with the exclusive-ownership publication invariant requested by review

### Network review

- CONNECT tunnel reuse is now origin-scoped, preventing protocol corruption from cross-origin tunnel reuse through the same proxy
- Proxy HTTP/1.1 requests now retry stale reused connections consistently with the direct transport path
- CONNECT requests now use the standard `Connection` header and forward-proxy requests always strip `Proxy-Authorization` when plaintext proxy auth is disabled
- CONNECT still uses HTTP/1.1 framing/ALPN in this path; HTTP/2-over-proxy remains explicitly deferred to Phase 22c
- Physical-device TLS validation is still required for iOS/Android IL2CPP before Phase 22b can be considered fully closed
