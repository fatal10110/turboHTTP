# Phase 22a.5 Step 6: Streaming File Downloader

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 6  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the file-download slice of 22a.5 so `FileDownloader` now uses the streaming client path instead of buffering the entire response in memory:

1. Rewrote `FileDownloader.DownloadFileAsync(...)` onto `SendStreamingAsync(...)`.
   - downloads now open `UHttpStreamingResponse`
   - response bytes are read incrementally from `response.Body`
   - bytes are written directly to a `FileStream` as they arrive
   - the response is always disposed asynchronously in a `finally` block

2. Added pooled-buffer streaming writes and incremental progress.
   - download transfer now uses a pooled 32 KiB buffer
   - progress callbacks fire per chunk instead of once at the end
   - `TotalBytes` remains `0` when the server does not provide `Content-Length`
   - download speed now reflects bytes transferred during the current operation, not any pre-existing resumed bytes already on disk

3. Preserved resume behavior on the streaming path.
   - existing partial files still trigger a `Range` request when resume is enabled
   - HTTP 416 still deletes the partial file and retries from scratch
   - HTTP 200 fallback after a resume attempt still restarts from scratch
   - mismatched `Content-Range` responses still discard the partial file and write the streamed body from the beginning

4. Moved checksum verification to incremental hashing.
   - `FileDownloader` now maintains `IncrementalHash` state while writing streamed bytes
   - when resuming with checksum verification enabled, the existing partial file is first fed into the incremental hash so the final digest still covers the full file
   - MD5 and SHA256 verification remain supported

5. Documented the current `Expect: 100-continue` limitation on streaming request bodies.
   - added XML documentation warnings to `StreamRequestBody` and `FactoryRequestBody`
   - the docs now explicitly note that non-replayable uploads can be rejected only after bytes have already started sending

## Decisions / trade-offs

1. **Checksum verification is incremental but still whole-file correct for resumed downloads.**  
   Rather than hashing the final file in a second pass, the downloader seeds the incremental hash from the existing partial file when resume + checksum verification are both enabled. This keeps the streaming write path bounded while preserving checksum correctness.

2. **Chunk progress is reported from the download loop, not from transport internals.**  
   `FileDownloader` remains a thin consumer of `SendStreamingAsync(...)`; it does not add transport-specific hooks just for progress. Progress therefore reflects bytes successfully written to disk.

3. **Unknown total size reports `0` rather than a misleading partial total.**  
   When `Content-Length` is absent, the downloader no longer reports `existingSize` as if it were the final total.

4. **The request-body warning was documented, not implemented.**  
   `Expect: 100-continue` remains explicitly deferred to post-22a. This step only documents the limitation on the streaming request-body types.
