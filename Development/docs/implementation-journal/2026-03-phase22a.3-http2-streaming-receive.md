# Phase 22a.3 Step 2-8: HTTP/2 Streaming Receive Path

**Date:** 2026-03-19  
**Phase:** 22a.3 Steps 2-8  
**Status:** Implemented and validated in a local HTTP/2 runtime harness

## What was implemented

Completed the remaining HTTP/2 streaming transport work for Phase 22a.3:

1. Replaced the old "buffer full response in `Http2Stream` and invoke handler at `END_STREAM`" model with an immediate response-start path:
   - initial response HEADERS now create an `Http2ResponseBodySource`
   - `IHttpHandler.OnResponseStartAsync(...)` is invoked as soon as initial headers are decoded
   - buffered collectors and streaming collectors now share the same HTTP/2 transport body source
2. Added a bounded per-stream receive body source backed by `SingleReaderChannel<Http2ResponseBodyChunk>`:
   - body DATA is copied into pooled frame-sized chunks
   - queue capacity is byte-bounded by `max(256 KB, local INITIAL_WINDOW_SIZE, local MAX_FRAME_SIZE)`
   - the read loop never blocks on a full queue; overflow faults the stream and sends `RST_STREAM(FLOW_CONTROL_ERROR)`
3. Implemented the decoupled HTTP/2 flow-control model:
   - connection-level receive credit is replenished independently of consumer reads, subject to an aggregate buffered-bytes cap
   - per-stream receive credit is replenished only when the consumer fully drains a queued DATA-frame chunk
   - aggregate buffered bytes are tracked across all streams so connection-level `WINDOW_UPDATE` can be deferred when buffered memory exceeds the internal 8 MB cap
4. Added `Http2ResponseBodySource`:
   - `ReadAsync` is non-buffering and non-destructive under caller cancellation
   - `GetTrailersAsync` asynchronously waits for trailing HEADERS completion
   - zero-body `HEADERS+END_STREAM` responses are created in a pre-completed state
5. Added post-reset DATA handling:
   - bytes for recently reset streams still decrement the connection receive window
   - connection-level `WINDOW_UPDATE` can still be emitted for those bytes
   - redundant `RST_STREAM` is suppressed for recently reset streams
6. Added the HTTP/2 abort / early-dispose protocol:
   - disposing a streaming response releases queued pooled buffers
   - active streams are reset with `RST_STREAM(CANCEL)`
   - further producer writes are discarded once the body source is aborted
7. Added coarse-grained stall detection:
   - active streams with buffered unread DATA and no consumption for the configured timeout are aborted
   - the abort path releases queued buffers and resets the stream
8. Added trailer completion support for streaming HTTP/2 responses.

## Decisions and trade-offs

1. **HTTP/2 response DATA is queued per frame, not via a giant contiguous buffer.**
   This lets the read loop keep a strict non-blocking contract while preserving stream-level flow-control accounting at frame granularity, including padding credit on the final consumption of a frame.
2. **Stall detection uses a connection-level maintenance loop rather than piggybacking only on inbound frame reads.**
   This is a small deviation from the phase text, but it preserves the "no per-stream timer" requirement and keeps stall detection active even when the peer stops sending frames after filling the buffer.
3. **The stall timeout field is internal and mutable (not readonly) for validation.**
   No public API changed; this only allows targeted tests/harnesses to lower the timeout without waiting a full minute.
4. **Aggregate memory capping is internal for 22a.3.**
   The 8 MB connection cap and 256 KB per-stream default are still transport-internal constants for now. Exposing them through `StreamingOptions` remains the planned 22a.4 follow-up.
