# Phase 22a.4 Step 2: Direct Buffered Request-Body Send Path

**Date:** 2026-03-20  
**Phase:** 22a.4 Step 2  
**Status:** Implemented and validated in a local compile/run harness

## What was implemented

Completed the next validated Phase 22a.4 slice by finishing the missing HTTP/1.1 buffered-request fast path while confirming the existing HTTP/2 buffered send path still satisfies the Step 2 contract:

1. Added threshold-aware HTTP/1.1 coalescing for small buffered request bodies.
   - `Http11RequestSerializer` now reads `StreamingOptions.SmallBufferedRequestThresholdBytes`
   - when a request body is already buffered and its length is within the threshold, the serializer appends the body bytes directly into the pooled header writer and performs a single stream write for `headers + body`
   - this preserves the no-session, no-stream-adapter buffered path while avoiding the extra body write call for small JSON/form payloads

2. Kept large buffered HTTP/1.1 bodies on the direct memory path without opening a read session.
   - bodies above the threshold still bypass `RequestBodyReadSession`
   - headers and body are written separately, but the body still goes directly from `ReadOnlyMemory<byte>` to the stream
   - no chunked framing is introduced for buffered bodies because `Content-Length` remains known

3. Confirmed the HTTP/2 buffered request-body path remains compliant with Step 2.
   - `Http2Connection.Send.cs` already sends buffered request bodies directly from `ReadOnlyMemory<byte>` into DATA frames
   - no streaming session is opened for buffered bodies
   - the validation harness reran an existing buffered POST HTTP/2 regression test to confirm the fast path still behaves correctly after the surrounding Step 1 option plumbing

4. Added focused serializer coverage for the new threshold behavior.
   - small buffered HTTP/1.1 body within threshold now asserts a single write
   - larger buffered HTTP/1.1 body above threshold now asserts separate header/body writes
   - both tests still verify payload correctness after serialization

## Decisions and trade-offs

1. **Small HTTP/1.1 buffered bodies are coalesced into the existing pooled header writer.**  
   This avoids adding a second transfer buffer just to reach a single write call. The body bytes are copied once into the already-rented pooled writer only on the small-body fast path where the threshold explicitly allows that trade-off.

2. **Large buffered HTTP/1.1 bodies stay on the direct two-write path.**  
   For bodies above the threshold, the serializer still avoids session creation and transfer-buffer rental, but it does not copy the full body into the header writer just to collapse the write count. This keeps larger buffered requests on the lower-copy path.

3. **HTTP/2 did not require a behavioral change for this step.**  
   The existing buffered-body path already matched the Step 2 requirements: direct memory-to-DATA-frame send, no `RequestBodyReadSession`, and no transfer-buffer rental. Step 2 validation therefore treated HTTP/2 as a regression check rather than a new implementation area.
