# Phase 22a.5 Step 2: Retry Drain / Replayability Rewrite

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 2  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the retry-specific 22a.5 rewrite so the interceptor no longer relies on buffered-era transport behavior:

1. `RetryInterceptor` now gates retry eligibility on request-body replayability.
   - requests with `request.Content.Replayability == RequestBodyReplayability.NonReplayable` no longer enter the retry loop
   - this applies even for idempotent methods
   - replayable buffered bodies and `ReplayableViaFactory` bodies remain eligible

2. `RetryDetectorHandler` now explicitly discards retryable 5xx response bodies through `IResponseBodySource`.
   - removed the old `TransportBehaviorFlags.SelfDrainsResponseBody` requirement
   - on retryable 5xx, the detector:
     - calls `body.DrainAsync(...)`
     - uses a linked cancellation source with a 2-second discard budget
     - falls back to `body.Abort()` if drain fails or times out
     - finalizes cleanup with `DisposeAsync()`

3. Retry behavior remains conservative once a response is committed downstream.
   - post-response-start body read failures still surface to the consumer without retry
   - retryable exception handling before commitment remains unchanged

## Decisions / trade-offs

1. **Replayability is now a hard gate before entering the retry loop.**  
   This keeps non-replayable requests from being partially processed by retry scheduling logic at all.

2. **The response discard timeout is fixed at 2 seconds for this step.**  
   This matches the repo’s existing short drain/cleanup posture and avoids letting retry suppression stall for an entire request timeout window. If mobile/device validation shows this is too aggressive, tune in 22a.6.

3. **Drain failure does not block retry.**  
   If the 5xx body cannot be drained cleanly, the detector aborts the body source and still allows the retry path to proceed. The retry decision is based on the response status and request replayability, not on successful best-effort cleanup.

  - retryable 5xx responses
  - retryable transport exceptions

- retryable 5xx suppression aborts the body before retry when drain fails
- errors that occur after a successful response-start commit still do not retry
