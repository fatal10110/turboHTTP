# Phase 22a.2 Step 3: HTTP/1.1 Request Retry / Failure Semantics

**Date:** 2026-03-18  
**Phase:** 22a.2 Step 3  
**Status:** Implemented and validated in a runtime harness; Unity/package validation still pending

## What Was Implemented

Completed the committed-body-aware retry slice for HTTP/1.1 request streaming:

1. `Http11RequestSerializer` now exposes per-attempt write telemetry through `Http11RequestWriteState`.
2. The serializer marks the request as body-committed when the first request-body write starts:
   - direct buffered-body fast path
   - known-length streamed bodies
   - chunked streamed bodies
3. `RawSocketTransport` now distinguishes between:
   - replayable requests where no request body bytes started sending yet
   - replayable requests whose body already started sending
   - non-replayable requests whose body already started sending
4. HTTP/1.1 stale reused-connection retry logic now allows replayable non-idempotent bodies to retry only when the body had not started sending yet.
5. When a non-replayable body has started sending and transport I/O fails, the request now fails immediately with a dedicated replayability-aware network error message instead of a generic stale-retry path.
6. Added transport regressions covering:
   - replayable `POST` retry on stale reused connection before body commitment
   - replayable `POST` no-retry after body bytes start on a reused connection
   - non-replayable body failure after send-start surfaces the dedicated error

## Decisions / Trade-Offs

1. **Commit tracking is intentionally boolean for Step 3.** The retry decision only needs to know whether body transmission started, not an exact byte count.
2. **HTTP/1.1 retry rules are now broader than the old idempotent-only gate, but only before body commitment.** Replayable `POST` / `PATCH` style requests can recover from a stale reused socket when the failure happens before any body write starts.
3. **The committed non-replayable failure remains a `NetworkError` category.** This keeps the existing public error model intact while still surfacing a transport-specific message that explains why the request cannot be retried.
4. **HTTP/2 retry behavior stays conservative in this step.** The new committed-body tracking is only wired into the HTTP/1.1 request serializer path; HTTP/2 keeps the existing replayable-idempotent retry gate until 22a.3 work.
