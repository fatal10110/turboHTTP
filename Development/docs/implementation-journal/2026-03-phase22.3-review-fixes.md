## What Was Fixed

This pass addressed the open findings from `Development/docs/phases/phase22/phase-22.3-implementation-review.md`:

1. Restored the missing `System.Threading` imports in `AdaptiveInterceptor.cs` and `BackgroundNetworkingPolicy.cs` so the Core assembly compiles again under Unity 2021.3 / C# 8.
2. Fixed `RetryInterceptor` terminal-attempt telemetry so `RetryExhausted` is only recorded when the last attempt actually ends in a retryable failure path; successful terminal attempts now record `RetrySucceeded` instead.
3. Moved cache-store body snapshotting and cache-entry preparation off the synchronous `OnResponseEnd(...)` path by handing `SegmentedBuffer` ownership to a detached background store task after the inner handler completes.
4. Implemented cache `stale-while-revalidate` support, including stored SWR metadata on `CacheEntry`, immediate stale serving, request cloning, a fresh background `RequestContext`, and fire-and-forget conditional revalidation.
5. Fixed `MemoryCacheStorage` expiry sweeping so entries that are expired but still inside their valid stale-while-revalidate window are retained instead of being purged immediately.
6. Added targeted runtime regressions covering terminal retry telemetry, stale-while-revalidate background context usage, and the ordering change that keeps cache storage from starting before the inner `OnResponseEnd(...)` callback returns.

## Decisions / Trade-Offs

1. **Cache uses a friend-assembly hook instead of widening the public Core API:** `RequestContext.CreateForBackground(...)` and `RequestContext.Clear()` remain internal; Cache accesses them via `InternalsVisibleTo("TurboHTTP.Cache")` rather than turning background-context lifecycle into public API.
2. **Background cache storage no longer writes timeline events after request completion:** the previous `CacheStore` event depended on the foreground `RequestContext`, which is cleared by `UHttpClient.SendAsync(...)`. The fix prioritizes correctness and completion-path latency over preserving that unsafe event write.
3. **Terminal retry telemetry now mirrors non-terminal semantics:** the new terminal observer tracks whether the final attempt committed a non-retryable response or delivered a retryable failure so the last-attempt path no longer reports a successful recovery as exhaustion.
