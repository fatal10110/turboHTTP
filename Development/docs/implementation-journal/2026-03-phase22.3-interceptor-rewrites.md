# Phase 22.3: Interceptor Rewrites

**Date:** 2026-03-09  
**Phase:** 22.3 (module interceptor rewrites)  
**Status:** Runtime module migration implemented; full Unity compile/device validation still pending

## What Was Implemented

Phase 22.3 migrated the remaining runtime pipeline modules from the deleted middleware contract to the interceptor/handler model introduced in 22.1/22.2.

The main runtime changes were:

1. Replaced the module-level `*Middleware` runtime classes with `*Interceptor` implementations across Auth, Middleware, Observability, Retry, RateLimit, Cache, and the remaining Core-owned adaptive/background pieces.
2. Split response-observing behavior into dedicated handler companions where Phase 22 requires handler wrapping: logging, metrics, monitor capture, adaptive sampling, cookie persistence, retry detection, redirect chaining, cache storage, and decompression.
3. Added `DecompressionInterceptor` / `DecompressionHandler` and the new core `ReadOnlySequenceStream` adapter so gzip/deflate responses can be transparently decoded without copying the compressed body into a `MemoryStream`.
4. Split `BackgroundNetworkingInterceptor` out of `BackgroundNetworkingPolicy.cs` so policy/config remains declarative and replay queue behavior stays on the interceptor, including `TryDequeueReplayable(out UHttpRequest request)`.
5. Updated the affected runtime/editor tests to the interceptor API and added a test-only `HttpPipeline` compatibility adapter in `Tests/Runtime` so the remaining 22.4-era test surface can execute against `InterceptorPipeline` without reintroducing runtime compatibility shims.

## Decisions / Trade-Offs

1. **Cache revalidation/invalidation use a module-local buffered collector:** `DispatchBridge` is internal to `TurboHTTP.Core`, so `CacheInterceptor` uses a local buffered collector for conditional revalidation and unsafe invalidation side-paths rather than piercing asmdef boundaries. Cache hits still short-circuit via direct handler callbacks, and cache misses still stream through `CacheStoringHandler`.
2. **Logging now emits request + response-start + response-end lines:** the interceptor/handler split makes response lifecycle visibility explicit. Tests were updated to the new line count instead of preserving the old single buffered response log line.
3. **Monitor capture stays handler-driven, not transport-buffered:** `MonitorHandler` builds a temporary buffered response snapshot only for monitor/event creation, after forwarding callbacks to the real downstream handler.
4. **Background policy and behavior are now explicitly split:** queue state and replay APIs live on `BackgroundNetworkingInterceptor`, while `BackgroundNetworkingPolicy` remains configuration-only.
5. **Decompression remains gzip/deflate-only in Phase 22.3:** Brotli stays deferred because Unity 2021.3 Mono availability still needs the separate platform probe planned in later phases.
6. **Test compatibility is scoped to the test assembly only:** the new `Tests/Runtime/Compatibility/HttpPipeline.cs` adapter keeps the runtime clean-break intact while reducing churn before Phase 22.4’s broader test migration is complete.

### Findings Addressed

| Rubric | Finding | Resolution |
|--------|---------|------------|
| Infrastructure | Request-mutating interceptors must obey clone-on-write because `UHttpRequest.WithHeader()`/`WithMetadata()` mutate in place | Auth, DefaultHeaders, Cookie, and adaptive timeout rewrites clone before mutating and call `RequestContext.UpdateRequest(...)`. |
| Infrastructure | Background queue/replay state should not remain hidden inside policy/config types | `BackgroundNetworkingInterceptor` was split into its own file/class and retained the public `TryDequeueReplayable(...)` API. |
| Infrastructure | Optional modules must not reach into Core internals casually | Cache revalidation/invalidation uses a local collector instead of depending on internal `DispatchBridge`, preserving assembly boundaries. |
| Network | Retry must follow the Phase 22 error-delivery contract (callback-based errors, cancellation via exception) | `RetryInterceptor` now relies on `RetryDetectorHandler.OnResponseError(...)` / 5xx `OnResponseStart(...)` instead of catching `UHttpException` from `await next(...)`. |
| Network | Redirect chains must keep the outer dispatch task pending until the terminal hop completes | `RedirectHandler` now bridges spawned redirect tasks into a `Completion` TCS so the outer interceptor remains pending across the full chain. |
| Network | Decompression must avoid LOH-heavy `MemoryStream` copies and must always return rented buffers on error | Added `ReadOnlySequenceStream`, used `SegmentedBuffer` for compressed-body staging, and returned the 64 KB `ArrayPool<byte>` rent in a `finally` block. |
