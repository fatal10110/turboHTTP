# Phase 22a.5 Step 3: Redirect Drain / Replayability Rewrite

**Date:** 2026-03-21  
**Phase:** 22a.5 Step 3  
**Status:** Implemented and validated in the local `dotnet` harness

## What was implemented

Completed the redirect-specific 22a.5 rewrite so redirect following no longer depends on buffered-body assumptions:

1. Redirect response suppression now cleans up the original response body through `IResponseBodySource`.
   - `RedirectHandler` now uses a 2-second discard budget for suppressed redirect responses
   - on redirect follow-up it calls `body.DrainAsync(...)`
   - if drain fails or times out, it falls back to `body.Abort()`
   - cleanup is finalized with `DisposeAsync()`

2. Redirect replayability is now enforced against the request-body model.
   - body-preserving redirects reject `RequestBodyReplayability.NonReplayable`
   - body-dropping redirects remain allowed even for non-replayable request bodies
   - replayable seekable streams are preserved without forced buffering

3. Redirect request rebuilding now respects the 22a content model.
   - buffered and cloneable request bodies still use detached copies for follow-up requests
   - replayable `StreamRequestBody` instances use shared-content follow-up requests so the stream can be reopened without forcing a full buffer
   - redirect timeout enforcement now mutates the already-built follow-up request timeout instead of rebuilding the request body again

4. Downstream response-start failures are preserved.
   - if a downstream `OnResponseStartAsync` throws on a non-redirect terminal response, `RedirectHandler` now records that failure into its completion task before rethrowing
   - this prevents swallowed downstream callback failures from being replaced by a synthetic "terminal callback" error

## Decisions / trade-offs

1. **Buffered bodies still detach-copy across redirect hops.**  
   This preserves the existing ownership isolation for byte-array-backed request bodies while allowing streaming-safe behavior for replayable stream bodies.

2. **Replayable seekable streams use shared request-body ownership instead of forced cloning.**  
   `StreamRequestBody` intentionally cannot detached-clone, so follow-up redirect requests share the existing replayable stream body and rely on its reopen semantics. The original request remains the owning lifetime root.

3. **Redirect cleanup uses the same bounded 2-second discard posture as retry cleanup.**  
   This keeps redirect suppression from stalling for the full request timeout while still giving the transport a short best-effort drain window. Device tuning, if needed, remains a 22a.6 validation item.

4. **Timeout budgeting now mutates the already-built redirect request.**  
   This avoids rebuilding request content a second time, which is especially important once redirects support replayable stream bodies.
