# Phase 22.1: Review Fixes

**Date:** 2026-03-08  
**Phase:** 22.1 follow-up (implementation review fixes)  
**Status:** Core review findings fixed; broader 22.3/22.4 migration debt still remains outside this pass

## What Was Fixed

This pass addressed the 22.1 review items inside the implemented core/testing slice:

1. Removed `TurboHTTP.Testing` dependence on Core internals by introducing a public `TransportDispatchHelper` wrapper and removing `InternalsVisibleTo("TurboHTTP.Testing")`.
2. Rewrote the stale editor test that still referenced deleted `HttpPipeline` / `IHttpMiddleware` APIs by replacing the body with an explicit phase-tracked ignore instead of a broken compile-time dependency.
3. Consolidated `UHttpClient.SendAsync` onto `DispatchBridge.CollectResponseAsync`.
4. Reworked plugin capability enforcement to remove the captured-local `Interlocked` path, eliminate the `GuardedNext` local-function allocation, reduce response-monitor allocations, and stop misclassifying pass-through transport failures as unauthorized error handling.
5. Implemented frozen `HttpHeaders.Empty` and switched callback delivery back to that safe singleton.
6. Removed the `emitRequestStart` split path by requiring transports to call `OnRequestStart` directly before `DeliverResponse`.
7. Made `RawSocketTransport.SendAsync` internal.
8. Hardened `ResponseCollectorHandler` by disposing buffered bodies on incomplete completion, lazily allocating the segmented buffer, and synchronizing buffer lifetime against cancellation/error completion.
9. Decoupled buffered collector/background-queue transport state from magic dispatch-state keys by moving cancellation/response-error storage onto dedicated `RequestContext` helpers and a public transport-facing cancellation hook on `TransportDispatchHelper`.
10. Tightened `InterceptorPipeline` validation (null elements now throw) and removed unnecessary pipeline rebuild work in `UHttpClient.Dispose()`.
11. Split the disabled fast-path in `AdaptiveInterceptor` so it no longer allocates an async state machine when adaptation is off.
12. Follow-up re-review fixes: switched tests from `DispatchBridge` to `TransportDispatchHelper`, fixed the pass-through plugin fault-ordering bug, documented exact-type cancellation handling in the collector, aligned transport/mock cancellation callback delivery with `OnResponseError`, made plugin shutdown on `Dispose()` best-effort/non-blocking, split the disabled fast-path in `BackgroundNetworkingInterceptor`, widened adaptive response-byte tracking to `long`, collapsed read-only plugin response monitoring down to an invocation guard plus observed handler, and stopped cloning `PluginContext.OptionsSnapshot` on every access.
13. Final pass-3 follow-up fixes: guarded `InvocationGuard` against post-release response recording, normalized concurrent dispose races on `_pluginLifecycleGate`, aligned `RecordReplayTransport.DispatchAsync` cancellation callbacks with the other transports, documented the `OnRequestStart` precondition on `TransportDispatchHelper.DeliverResponse`, removed the last manual response-delivery loop from the editor test transport, replaced the generic interceptor-presence reflection path with `interceptor is T`, and added a full-trust plugin fast path that skips the guard wrapper.

## Decisions / Trade-Offs

1. **Public shim instead of widening `DispatchBridge`:** shipping assemblies now depend on `TransportDispatchHelper`, while `DispatchBridge` remains internal transitional scaffolding for the core slice.
2. **Frozen empty headers instead of per-call allocation:** callback delivery can safely reuse `HttpHeaders.Empty` now that mutation is rejected.
3. **Invocation-guard monitoring instead of separate monitor + forwarding handler objects:** read-only plugin monitoring now keeps its comparison queue inside `InvocationGuard`, rents comparison storage from `ArrayPool<T>`, and allocates only the guard plus per-dispatch observed handler on the hot path.
4. **Unauthorized error handling is only flagged when a downstream fault was actually handled:** pass-through transport errors are no longer blamed on read-only observer plugins.
5. **Cancellation now reaches both buffered callers and handler observers:** transports store the `OperationCanceledException` through `TransportDispatchHelper` for buffered collection while still issuing `OnResponseError(UHttpException Cancelled, ...)` for interceptor/observer visibility.
6. **`Dispose()` no longer blocks on plugin shutdown:** shutdown is now best-effort and asynchronous during disposal, avoiding main-thread stalls and deadlock risk. Explicit `UnregisterPluginAsync` remains the graceful shutdown path.
7. **`ResponseCollectorHandler` remains a buffered bridge for 22.1:** the concurrency hardening here is enough for the current bridge path, but the full streaming/ownership model still belongs to 22.2.
8. **`PluginContext.OptionsSnapshot` is now a stable plugin-local object rather than a clone-per-access defensive copy:** plugins can still mutate only their isolated snapshot, while client options remain unaffected.
9. **Record/replay passthrough triple-buffering remains intentional for now:** this is test-only infrastructure and is documented here rather than optimized in the 22.1 review pass.
10. **Read-only plugin observation now rejects post-scope callbacks explicitly:** this closes the ArrayPool fragility before 22.2 streaming makes late callback hazards realistic.
11. **Fully capable plugins skip the capability-enforcement wrapper:** when a plugin already has request mutation, response mutation, error handling, and redispatch rights, the extra guard allocation no longer adds value.
