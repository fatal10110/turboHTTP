# Phase 19a.5 — Internal HTTP Object Pooling

**Date:** 2026-02-27
**Status:** COMPLETE (2 review rounds, all findings resolved)

---

## What Was Implemented

Phase 19a.5 adds internal object pooling for four types to reduce per-request GC pressure. `UHttpResponse` was intentionally kept non-pooled (Step 0) because it escapes to user code.

### Step 1: ParsedResponse Pool

`ParsedResponse` is a short-lived carrier struct used to transfer parsed HTTP/1.1 response data from `ParseAsync` to `RawSocketTransport.BuildResponse`. Previously allocated fresh per request; now rented from a bounded pool.

- Added `Reset()` to `ParsedResponse` clearing all 6 fields to sentinel values
- Created `ParsedResponsePool` using `ObjectPool<ParsedResponse>` with capacity 16
- `ParseAsync` rents from pool at the success return point using an **ownership-transfer pattern** (see below)
- `RawSocketTransport.BuildResponse` returns to pool after extracting data into `UHttpResponse`

**Ownership-transfer pattern** (to handle exception paths safely):
```csharp
ParsedResponse parsedResult = null; // declared before try block
// ... inside try ...
parsedResult = ParsedResponsePool.Rent();
// ... populate fields ...
var toReturn = parsedResult;
parsedResult = null;       // transfer ownership; finally won't double-return
return toReturn;
// finally block:
if (parsedResult != null)  // only non-null if exception fired after Rent() but before null
    ParsedResponsePool.Return(parsedResult);
```

### Step 2: Http2Stream Pool

`Http2Stream` holds a `MemoryStream` (`HeaderBlockBuffer`) for header block accumulation. The MemoryStream is expensive to allocate and has no native resources, so it can safely be reused across requests.

**Refactoring approach:**
- Parameterless constructor: allocates `MemoryStream` once
- `Initialize(streamId, request, context, sendWindow, recvWindow, responseSourcePool)`: sets all per-request state
- `PrepareForPool()`: resets without disposing `MemoryStream` — returns `_responseBodyBuffer` to `ArrayPool`, disposes `CancellationRegistration`, clears all references
- `Http2StreamPool`: `ObjectPool<Http2Stream>` with capacity 128 (above MAX_CONCURRENT_STREAMS default of 100); `Rent()` calls `Initialize()`; `Return()` calls `PrepareForPool()` via the reset delegate

**Key lifecycle rule:** The read loop must **never** call `stream.Dispose()` after signalling a stream via `Fail()`/`Complete()`/`Cancel()`. `SendRequestAsync`'s `finally` block is the sole owner of `Http2StreamPool.Return(stream)`. This rule was enforced by removing all 13 `stream.Dispose()` calls from the read loop.

**`PrepareForPool` memory ordering:** `Volatile.Write(ref _disposed, 0)` and `Volatile.Write(ref _responseCompleted, 0)` are placed at the very end of `PrepareForPool()`, after all field clears, establishing a release fence on ARM IL2CPP. Reads via `ThrowIfDisposed()` and `IsResponseCompleted` use `Volatile.Read`, forming a correct acquire/release pair.

### Step 3: Header Parse Scratch Pool

HTTP/1.1 header parsing accumulated raw name-value pairs into a temporary list before building `HttpHeaders`. Previously, a new `List<(string, string)>` was allocated per request; now a pooled `HeaderParseScratch` object is rented.

- `HeaderParseScratch`: holds `List<(string Name, string Value)> RawHeaders` (initial capacity 16)
- `HeaderParseScratchPool`: `ObjectPool<HeaderParseScratch>` with capacity 16 in `TurboHTTP.Core.Internal`
- `ParseAsync`: rents scratch before the `do-while` loop; clears per iteration; builds `new HttpHeaders()` from scratch at end; returns scratch in `finally`
- String references transferred to `HttpHeaders` are heap-allocated and GC-managed — clearing the scratch list's backing array slots is safe

### Step 4: HpackEncoder Writer State

`HpackEncoder.Encode()` previously allocated a new `PooledByteBuffer` per call (which itself managed an `ArrayPool<byte>` buffer internally, incurring rent/return overhead per encode).

- Added `_outputBuffer = new PooledByteBuffer(256)` as instance field on `HpackEncoder`
- Added `Reset()` to `PooledByteBuffer`: sets `_length = 0` without returning backing array to `ArrayPool`
- `Encode()` calls `_outputBuffer.Reset()` then reuses it — backing array grows as needed and is retained for the encoder's lifetime
- `HpackEncoder` implements `IDisposable`; `Dispose()` returns `_outputBuffer`'s backing array to `ArrayPool`
- `Http2Connection.Dispose()` calls `_hpackEncoder?.Dispose()` for clean teardown

## Decisions Made

### Read loop owns signalling only; SendRequestAsync owns pool return
The single most important architectural decision: the read loop only signals streams (`Fail`, `Complete`, `Cancel`). Pool return is owned exclusively by `SendRequestAsync`'s `finally` block, which runs asynchronously on a thread pool thread after the `ValueTask` continuation fires (`RunContinuationsAsynchronously = true`). This prevents the critical defect where `stream.Dispose()` called from the read loop would destroy the `MemoryStream` before `PrepareForPool()` could run.

### MemoryStream reuse across requests
`Http2Stream.HeaderBlockBuffer` is a `MemoryStream` that grows to accommodate the largest header block seen and stays allocated for the stream's pool lifetime. When a stream is returned to the pool, `SetLength(0)` resets it without freeing the buffer. This eliminates per-request `MemoryStream` allocation at the cost of holding ~4KB per pooled stream instance.

### ParsedResponsePool in Transport, HeaderParseScratchPool in Core.Internal
`ParsedResponsePool` is an implementation detail of the Transport assembly's parsing path — it lives in `TurboHTTP.Transport`. `HeaderParseScratchPool` is used from `Http11ResponseParser` which imports `TurboHTTP.Core.Internal`; it lives in that namespace alongside `ObjectPool<T>` and `ByteArrayPool`.

### HpackEncoder._outputBuffer: connection-lifetime allocation
The `PooledByteBuffer` is kept alive for the lifetime of the `Http2Connection` rather than renting/returning per encode. This trades a small constant memory overhead (~256 bytes initial, grows to max header block seen) for zero per-encode allocation overhead. The buffer is returned to `ArrayPool` on connection disposal.

### Pool capacity choices
- `ParsedResponsePool`: capacity 16 — matches HTTP/1.1 serial request pattern (1 active parse at a time per connection)
- `HeaderParseScratchPool`: capacity 16 — same rationale
- `Http2StreamPool`: capacity 128 — provides headroom above the HTTP/2 default MAX_CONCURRENT_STREAMS of 100 for the common case of a single connection; note the pool is connection-global (static), so with multiple concurrent connections the effective per-connection capacity is lower

## Review Findings and Resolutions

### Round 1 (unity-infrastructure-architect + unity-network-architect)

Both reviewers independently identified the same root cause: **13 `stream.Dispose()` calls in the read loop** that contradicted the design intent stated in `FailAllStreams`'s comment. Additional findings:

| Severity | Finding | Resolution |
|---|---|---|
| Critical | `stream.Dispose()` in read loop after `Fail()/Complete()` destroys `MemoryStream` before `PrepareForPool()` | Removed all 13 `stream.Dispose()` calls from read loop |
| Critical | Same TOCTOU from `HandleGoAwayFrame` calling `stream.Dispose()` | Fixed as part of the above |
| Major | `_disposed = 0` plain store in `PrepareForPool` — ARM memory ordering | Changed to `Volatile.Write(ref _disposed, 0)` |
| Major | `ParsedResponsePool.Rent()` outside inner `finally` — fragile on future exception | Implemented ownership-transfer pattern |
| Major | Duplicate files `Runtime/Performance/HeaderParseScratchPool.cs` | False positive — those files were never created; pool lives only in Core.Internal |
| Minor | `stream.RecvWindowSize` WINDOW_UPDATE sent for disposed stream | Guarded with `!responseBodyDisposed` |

### Round 2 (verification)

Both reviewers confirmed all three fixes were correctly implemented with no new issues introduced. Residual pre-existing findings (low severity) documented for backlog:
- No RST_STREAM sent on header-decode error paths in `DecodeAndSetHeaders` (pre-existing; deferred)
- `CancellationRegistration.Dispose()` in `PrepareForPool` may block briefly on heavily loaded mobile (pre-existing; informational)
