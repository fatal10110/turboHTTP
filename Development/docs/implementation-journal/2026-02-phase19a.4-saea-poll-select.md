**Status:** Complete (two-round specialist review, all findings resolved)

## What Was Implemented

Phase 19a.4 introduces two additional socket I/O modes alongside the existing `NetworkStream` baseline:

- **SAEA (SocketAsyncEventArgs):** Zero-allocation async socket I/O using OS-native completion APIs (IOCP on Windows, kqueue on macOS/iOS, epoll on Linux/Android). Avoids per-operation `Task` allocation; the transport hot path is genuinely zero-alloc.
- **PollSelect:** Dedicated background poll thread using `Socket.Poll` + synchronous `Send`/`Receive`. Designed for IL2CPP platforms (consoles, some mobile targets) where SAEA runtime callbacks have historically been unreliable or unavailable.

The implementation covers:
- `SocketIoMode` enum added to `TurboHTTP.Core` (`ConnectionPoolOptions`)
- `SaeaCompletionSource` — reusable `IValueTaskSource<int>` bridge for SAEA events
- `SaeaSocketChannel` — duplex SAEA send/receive channel with in-flight safety
- `SaeaStream` — thin `Stream` adapter over `SaeaSocketChannel`
- `PollSelectSocketChannel` — poll-thread channel with reusable `PollSelectSource` and `AbortPendingOperations`
- `PollSelectStream` — thin `Stream` adapter over `PollSelectSocketChannel`
- `SaeaPlatformDetection` — platform detection + diagnostic logging
- `TcpConnectionPool` integration — switches on `SocketIoMode` to build the correct stream (both normal and BouncyCastle fallback paths)

## Architectural Decisions

### SocketIoMode lives in TurboHTTP.Core
The enum must be visible to both `TurboHTTP.Core` (where `ConnectionPoolOptions` lives) and `TurboHTTP.Transport` (where the pool lives). Placing it in Core avoids a circular dependency — Transport already references Core, and Core has zero external references. The enum itself has no Unity or socket dependencies.

### In-flight tracking (SAEA): volatile int + SpinWait
`SocketAsyncEventArgs` completion callbacks fire on the OS thread pool. If `Dispose()` calls `ArrayPool.Return` before the IOCP/kqueue/epoll completion fires, the buffer could be claimed by another operation while the OS is still writing into it — silent memory corruption. The fix is conservative: `_recvActive`/`_sendActive` are volatile int flags set to 1 before the SAEA call and cleared to 0 in the completion handler (before delivering the result to the `ValueTaskSource`). `Dispose()` spins on both flags before returning buffers. `SpinWait` is appropriate because the completion handler runs in microseconds on the OS thread pool.

### Concurrent-call guard (both modes)
Both SAEA and PollSelect channels support at most one concurrent `ReceiveAsync` and at most one concurrent `SendAsync`. A second concurrent call to the same direction throws `InvalidOperationException`. HTTP/1.1 is half-duplex (send, then receive) so the common case is never concurrent. HTTP/2 requires its own multiplex layer above this (not in scope here).

### PollSelectSource implements both IValueTaskSource<int> and IValueTaskSource
The receive direction returns `ValueTask<int>` (bytes read). The send direction returns void `ValueTask`. Without the non-generic `IValueTaskSource` interface, the send path would have to call `AsValueTask().AsTask()` which allocates a `Task` on every call — defeating the zero-allocation goal. Implementing both interfaces on a single `ManualResetValueTaskSourceCore<int>` field is the documented BCL pattern; the non-generic `GetResult` simply ignores the int value.

### PollSelect: indefinite send retry on WouldBlock
When the kernel send buffer is full (`SocketError.WouldBlock` or `sent == 0`), the correct behaviour is to wait for writability and retry — not to throw. A hard inner timeout would cause spurious failures under TCP congestion on mobile networks. The outer request timeout (managed by `RawSocketTransport` via `CancellationTokenSource.CancelAfter`) is the only deadline. The inner retry loop checks `_disposed` each iteration to unblock cleanly on disposal.

### PollSelect: WouldBlock after Poll returns true (no false EOF)
`Socket.Poll(SelectRead)` returning true followed by `Socket.Receive` returning `WouldBlock` is a TOCTOU spurious wakeup on non-blocking sockets. If we were to deliver 0 bytes, the HTTP/1.1 parser would interpret it as graceful EOF and close the connection. The fix: return `false` from `TryServiceRecv()` to loop back and retry.

### PollSelect: thread join with IsAlive guard
Under Android GC pauses, the poll thread may still be alive after a join timeout. Returning `ArrayPool` buffers while the thread might still be writing into them would corrupt later operations. The 500ms join timeout covers normal GC pauses; `IsAlive` check afterward ensures we only return buffers once the thread has actually exited. If the thread is still alive after 500ms (pathological case), the buffers are leaked as a deliberate safety trade-off: a small one-time allocation leak is preferable to memory corruption.

### volatile bool _disposed in stream adapters
`SslStream` reads `CanRead`/`CanWrite` from its own internal threads. On ARM64 IL2CPP, without `volatile`, a non-volatile store to `_disposed` on the owning thread might not be observed by the SslStream thread (no store-release/load-acquire guarantees). `volatile bool` provides the required memory model at negligible cost.

### BouncyCastle fallback path uses same SocketIoMode switch
The BouncyCastle reconnect path in `TcpConnectionPool` originally hardcoded `NetworkStream`. After the fix, both the normal path and the BouncyCastle fallback path use the same `switch (_socketIoMode)` block — ensuring SAEA and PollSelect modes work even when BouncyCastle TLS is active.

### SaeaPlatformDetection: no SaeaIsExpectedToBeneficial
An initial draft had a property intended to advise whether SAEA was expected to be beneficial. It was removed because: (1) it was always true for supported platforms, (2) the mode selection is a user configuration choice (`ConnectionPoolOptions.SocketIoMode`), not a runtime recommendation. The class now provides only `CompletionModelName` (diagnostics) and `LogSelectedMode` (one-time log at pool construction).

## Specialist Review Summary

### Round 1 — Blocking Findings and Resolutions

Both the **unity-infrastructure-architect** and **unity-network-architect** specialists reviewed the initial implementation and found blocking issues.

**Infrastructure Review — Blocking:**
| ID | Issue | Resolution |
|----|-------|------------|
| B-1 | `ArrayPool.Return` called in `Dispose()` before IOCP completion fires — memory corruption | Added `_recvActive`/`_sendActive` volatile int flags; `Dispose()` spins until both are 0 |
| B-2 | No concurrent-call guard — second `ReceiveAsync`/`SendAsync` overwrites source | `Interlocked.CompareExchange` CAS at start of each direction |
| B-3 | `goto` across variable declarations in `RunPollLoop` — IL2CPP exception table risk; also bypasses `_disposed` check | Restructured into `TryServiceRecv()`/`TryServiceSend()` helpers with no `goto` |
| B-4 | Hard 100ms send-retry timeout throws `SocketError.TimedOut` on normal TCP backpressure | Replaced with indefinite retry loop checking `_disposed` each iteration |
| B-5 | `new object[] { _socket }` in `ct.Register` — per-operation allocation | Direct socket capture: `ct.Register(static s => { ... }, _socket)` |
| B-6 | `_disposed` field not `volatile` in stream adapters — ARM64 IL2CPP data race | `volatile bool _disposed` in both `SaeaStream` and `PollSelectStream` |
| B-7 | 100ms thread join insufficient under Android GC pause — buffers returned on live thread | 500ms join + `if (!_pollThread.IsAlive)` guard before `ArrayPool.Return` |

**Infrastructure Review — Advisory (all resolved):**
| ID | Issue | Resolution |
|----|-------|------------|
| A-4 | `AsVoidTask()` allocates `Task` on every send | Implemented `IValueTaskSource` (non-generic) on `PollSelectSource`; `AsVoidValueTask()` returns `new ValueTask((IValueTaskSource)this, _core.Version)` |
| A-9 | BouncyCastle fallback hardcodes `NetworkStream` | Applied same `switch (_socketIoMode)` in BouncyCastle reconnect path |
| A-10 | Non-unique poll thread names | `static int s_threadCounter` + `Interlocked.Increment` for unique names |
| A-11 | `new PollSelectSource()` per operation | Stored as `readonly` fields, reset with `Reset()` |
| A-12 | `SaeaIsExpectedToBeneficial` unused and always true | Removed the property entirely |

**Network Review — Blocking:**

|----|-------|------------|
| B-1 | Buffer returned before poll thread exits (same as infra B-7) | 500ms join + `IsAlive` guard |
| B-2 | No concurrent-call guard (same as infra B-2) | CAS guard |
| B-3 | `goto` across variable declarations (same as infra B-3) | `TryServiceRecv()`/`TryServiceSend()` restructure |

### Round 2 — Verification

Both specialists confirmed **all fixes correct and complete** with no regressions. Two minor advisory observations (no code changes needed):
1. `SaeaSocketChannel._disposed` uses `Volatile.Read` pattern (intentional asymmetry vs. stream adapters; both are safe, just different idioms).
2. `Thread.Sleep(0)` vs `Thread.Yield()` — documented CPU trade-off of PollSelect mode; not a defect.

None. All blocking and advisory findings were resolved in this session.
