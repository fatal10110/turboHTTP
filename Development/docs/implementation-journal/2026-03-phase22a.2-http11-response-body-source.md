# Phase 22a.2 Steps 5-6: HTTP/1.1 Response Body Source and Early-Dispose Policy

**Date:** 2026-03-18  
**Phase:** 22a.2 Steps 5-6  
**Status:** Implemented and validated in a runtime harness; Unity/package validation still pending

## What Was Implemented

Completed the HTTP/1.1 streamed-response body source and the drain-or-close policy that sits on top of the staged response parser:

1. Added `Http11ResponseBodySource` as the direct HTTP/1.1 `IResponseBodySource` implementation.
2. The body source now covers the four HTTP/1.1 framing modes resolved by `Http11ResponseParser`:
   - `Empty`
   - `ContentLength`
   - `Chunked`
   - `ReadToEnd`
3. `Content-Length` responses expose a known `Length`, stream directly from the transferred `BufferedStreamReader`, and return the connection to the pool only after the final byte is consumed.
4. Chunked responses parse chunk headers inline, stream decoded data directly into the caller buffer, discard trailers internally, and surface `HttpHeaders.Empty` from `GetTrailersAsync(...)`.
5. Read-to-end responses stream until EOF and always close the connection instead of reusing it.
6. Early async disposal now uses the 22a gate:
   - only keep-alive responses are candidates for reuse
   - fixed-length responses drain only when remaining unread bytes are within the 64 KB budget
   - chunked responses drain only when the terminal chunk is reached within the same 64 KB decoded-body budget
   - read-to-end and `Connection: close` responses skip drain and close immediately
   - drain attempts are bounded by a 2 second timeout
7. Body-read cancellation and protocol failures now fault the source and close the connection instead of leaving a potentially reusable lease behind.
8. Fixed the terminal-state transition so a body that already completed cleanly stays in the completed state instead of being reclassified as aborted by a later close path.
9. Added direct HTTP/1.1 regressions covering:
   - small `Content-Length` early dispose drains and reuses the connection
   - large `Content-Length` early dispose closes the connection
   - chunked early dispose reuses the connection when the terminating chunk is reached within budget
   - `HEAD` with `Content-Length` exposes an empty body source
   - read-to-end framing closes the connection
   - `Connection: close` skips drain and closes immediately
   - user cancellation during `ReadAsync(...)` closes the connection and prevents reuse

## Decisions / Trade-Offs

1. **Chunked drain budgeting is measured in decoded body bytes.** The policy is about bounded response payload consumption before reuse, not exact wire-byte accounting including chunk metadata.
2. **User cancellation is a terminal transport event for HTTP/1.1 body sources.** Once a streaming read is canceled, the source closes the lease rather than trying to keep the connection alive through partially consumed framing state.
3. **`GetTrailersAsync(...)` remains completion-only and returns `HttpHeaders.Empty`.** The source drains to end if needed, but full HTTP/1.1 trailer parsing is still intentionally deferred by the phase plan.
4. **Reuse decisions are driven by the parser’s keep-alive result, not ad-hoc header checks in the body source.** That keeps framing/connection policy centralized in the parser and avoids duplicating HTTP/1.1 keep-alive rules.
