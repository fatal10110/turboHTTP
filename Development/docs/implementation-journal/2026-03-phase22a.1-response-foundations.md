# Phase 22a.1: Response Body Foundations

**Date:** 2026-03-17  
**Phase:** 22a.1 follow-up (response-side substrate)  
**Status:** In progress (public response abstractions added; handler/transport/client migration still pending)

## What Was Implemented

Added the next isolated 22a.1 slice on the response side inside `TurboHTTP.Core`:

1. Added public `IResponseBodySource` as the new response-body abstraction.
2. Added `ResponseBodyStream`, a thin non-seekable `Stream` adapter over `IResponseBodySource`.
3. Added `UHttpStreamingResponse` with:
   - `StatusCode`
   - `Headers`
   - `Body`
   - `GetTrailersAsync(...)`
   - both `IDisposable` and `IAsyncDisposable`
4. Added focused runtime tests covering:
   - known-length body reads
   - unknown-length `Length` behavior
   - synchronous dispose abort semantics
   - asynchronous dispose semantics
   - trailer retrieval
   - release-callback lifetime behavior

This slice intentionally does **not** change `IHttpHandler`, transports, or `UHttpClient` yet. Existing dispatch code still uses the old buffered callback contract until the next step.

## Decisions / Trade-Offs

1. **Sync dispose is an abort path, not a blocking drain path:** `UHttpStreamingResponse.Dispose()` calls `Abort()` on the underlying body source and completes local lifetime cleanup. It does not try to synchronously emulate `DisposeAsync()`.
2. **Async dispose owns the actual body-source disposal path:** `DisposeAsync()` delegates to `IResponseBodySource.DisposeAsync()` and still guarantees request-release callbacks in `finally`.
3. **`ResponseBodyStream` stays intentionally thin:** no read-ahead buffer, no seek support, and no extra ownership layer beyond forwarding reads/disposal to the parent streaming response.
4. **Leak detection stays advisory for now:** the response finalizer only emits a development-time debug warning. It does not attempt transport cleanup from the finalizer thread.
5. **Buffered compatibility is deferred to the next slice:** this step only defines the public substrate. The internal buffered body-source implementation and bridge rewrite land together with the handler contract migration.

### Findings Addressed In This Pass

| Rubric | Finding | Resolution |
|--------|---------|------------|
| Infrastructure | Response lifetime must keep ownership and disposal explicit | `UHttpStreamingResponse` centralizes body lifetime and release callbacks |
| Infrastructure | Stream adapter should not add avoidable buffering or hidden ownership layers | `ResponseBodyStream` is a thin forwarder over the body source |
| Network | Public body-source contract must separate abort, read, drain, and trailer access | Added `IResponseBodySource` with explicit `Abort`, `ReadAsync`, `DrainAsync`, and `GetTrailersAsync` methods |
| Network | Sync disposal must not fake safe recovery on connection-scoped sources | `Dispose()` is explicitly abort-based instead of trying to synchronously drain |

### Deferred Findings

| Rubric | Finding | Target |
|--------|---------|--------|
| Infrastructure | `IHttpHandler` still uses `OnResponseData` / `OnResponseEnd` callback delivery | Phase 22a.1 next step |
| Infrastructure | `UHttpClient.SendAsync` and `request.SendAsync` have not yet been split to buffered/streaming APIs | Phase 22a.1 later step |
| Network | No transport implementation produces `IResponseBodySource` yet | Phase 22a.1 next step, then 22a.2 / 22a.3 |
| Network | IL2CPP physical-device validation for `IAsyncDisposable` and `ValueTask<HttpHeaders>` remains outstanding | Phase 22a.1 step 0 / 22a.6 validation track |
