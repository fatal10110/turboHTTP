# Phase 22b.4: HTTP/1.1 + HTTP/2 Request Trailers

**Date:** 2026-03-22  
**Phase:** 22b.4 request trailers  
**Status:** Implemented and validated in focused local harnesses

## What was implemented

Completed the next Phase 22b slice by wiring request-trailer support through Core, HTTP/1.1 serialization, and HTTP/2 send paths.

1. Added request-trailer state to the request body model.
   - `UHttpRequestBody` now stores an internal trailer provider plus declared trailer names
   - detached body cloning now copies trailer metadata centrally from the base clone path
   - the change stayed additive and did not alter module boundaries or asmdefs

2. Added the builder surface on `UHttpRequest`.
   - introduced `WithRequestTrailers(IReadOnlyList<string>, Func<HttpHeaders>)`
   - request content now owns the trailer provider, while `UHttpRequest` keeps the `Trailer` declaration header synchronized with that content
   - replacing request content now clears the managed `Trailer` header when trailer support is no longer present

3. Implemented HTTP/1.1 request-trailer serialization.
   - HTTP/1.1 now rejects non-chunked request bodies with a trailer provider before any request bytes are written
   - chunked uploads now serialize the terminal `0\r\n`, filtered trailer fields, and final CRLF via the existing pooled header writer path
   - prohibited request trailers are filtered with the shared `TrailerFieldValidator`
   - trailer values now reuse the existing header-name/value validation path, including CRLF rejection

4. Implemented HTTP/2 request trailers.
   - request-body send paths now decide `END_STREAM` placement based on whether the trailer provider produces any actual post-filter trailers
   - known-length buffered/stream bodies now keep `END_STREAM` off the last DATA frame when trailing HEADERS are required
   - a trailing HEADERS frame with `END_STREAM` is now emitted for filtered non-empty request trailers
   - prohibited request trailers are filtered and trailer names are normalized to lowercase for HTTP/2
   - invalid request-trailer values trigger stream reset and surface through the dispatch path as `UHttpException` with the original validation exception preserved as `InnerException`

5. Added focused coverage for the new behavior.
   - Core request-trailer/body metadata coverage
   - request builder/header synchronization coverage
   - HTTP/1.1 serializer wire-format and validation coverage
   - HTTP/2 send-path frame sequencing and trailer filtering coverage

## Decisions and trade-offs

1. Kept trailer metadata on `UHttpRequestBody` and copied it in the base detached-clone path.
   - this avoided transport-only wrapper bodies and kept optional module request bodies working through the existing `InternalsVisibleTo("TurboHTTP.Files")` arrangement

2. Treated the `Trailer` declaration header as managed request state once `WithRequestTrailers(...)` is used.
   - the header is re-synchronized when headers or content are replaced so the request surface does not drift from the underlying body state

3. Reused the existing pooled HTTP/1.1 header writer for the terminal chunk/trailer section.
   - this keeps trailer serialization on the established Latin-1 + pooled-buffer path instead of introducing ad-hoc string assembly

4. Retained the existing HTTP/2 unknown-length EOF pattern.
   - unknown-length request bodies still use the existing explicit zero-length `END_STREAM` DATA frame when EOF is only discovered by a follow-up read
   - known-length/buffered bodies now place `END_STREAM` on the last DATA frame when no actual trailers remain after filtering
   - this keeps the 22a unknown-length framing behavior stable and avoids speculative one-chunk lookahead buffering

5. Accepted transport-surface exception mapping for HTTP/2 request-trailer failures.
   - serializer-only validation remains raw `ArgumentException` / `InvalidOperationException`
   - once the request is inside `Http2Connection.DispatchAsync`, failures are surfaced through the existing `UHttpException` transport error model, with the original validation exception preserved as `InnerException`
